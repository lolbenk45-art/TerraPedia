# Terraria Wiki 数据获取方案分析

## 文档目的

本文只基于以下两个在线入口分析如何获取 Terraria Wiki 数据，并尽量满足以下目标：

- 不依赖本地现有文件
- 尽量不漏数据
- 尽量不触发站点风控
- 不写实现代码，仅给出需求分析和采集策略

分析对象：

- `https://terraria.wiki.gg/api.php`
- `https://terraria.wiki.gg/zh/`

补充说明：

- 对 `https://terraria.wiki.gg/zh/` 的分析过程中，确认该中文站实际存在独立 API 入口 `https://terraria.wiki.gg/zh/api.php`
- 因此，若目标是完整获取中文 Wiki 数据，仅看页面 HTML 不够，必须把 `zh/api.php` 纳入正式方案

## 一句话结论

最合理的方案是：

- 以 `https://terraria.wiki.gg/zh/api.php` 作为中文页面内容的主数据源
- 以 `https://terraria.wiki.gg/api.php` 作为共享文件仓库和英文补充数据源
- 以 `https://terraria.wiki.gg/zh/` 页面 HTML 作为少量校验和复杂页面渲染补充来源，而不是主抓取面

换句话说：

**中文正文走中文 API，共享媒体走根站 API，页面 HTML 只做校验，不做主采集。**

---

## 一、站点结构判断

### 1. 中文站不是英文站的简单语言参数页

在线返回结果表明：

- 中文站拥有独立 `wikiid`：`terraria_zh`
- 中文站拥有独立 `scriptpath`：`/zh`
- 中文站拥有独立 `articlepath`：`/zh/wiki/$1`
- 中文站拥有独立 API：`/zh/api.php`

这意味着：

- 中文页面数据不能仅靠根站 `api.php` 完整获取
- 中文内容、中文页面修订、中文分类与中文模板关系，原则上都应从 `zh/api.php` 获取

### 2. 根站 API 仍然不能忽略

实测中，中文页面引用的某些图片在中文站文件库中并不是本地文件，而是共享自英文仓库：

- 中文站查文件时会返回 `imagerepository=terraria_en`
- 同时仍能拿到图片 URL

这说明：

- 只抓中文 API，不足以构成“完整 wiki 数据”
- 至少在文件和媒体层，根站 `api.php` 仍然是必要数据源

---

## 二、目标拆分

如果“获取 Terraria Wiki 数据”是一个正式需求，则建议把目标拆成三层，而不是把所有内容视为同一种网页抓取任务。

### 1. 页面内容层

目标包括：

- 中文页面标题
- 页面正文原始内容
- 页面修订版本
- 页面最后更新时间
- 页面所属命名空间
- 重定向关系
- 页面分类、模板、内部链接、跨语言链接

### 2. 渲染结果层

目标包括：

- 页面实际渲染后的 HTML
- 模板展开后的结构
- 表格、图标、信息框等页面最终展示结果

### 3. 文件与媒体层

目标包括：

- 图片与动图 URL
- 文件元数据
- 文件哈希
- 文件大小、类型、尺寸
- 文件说明页来源
- 文件实际归属仓库

这三层的数据来源、稳定性和风控风险不一样，应该区别处理。

---

## 三、最合理的数据来源分工

## 1. 中文页面主数据源：`https://terraria.wiki.gg/zh/api.php`

这是主数据源，应承担：

- 页面枚举
- 页面正文获取
- 页面版本同步
- 页面关系同步
- 中文站最近变更追踪

适合承载的模块包括：

- `allpages`
- `recentchanges`
- `revisions`
- `info`
- `categories`
- `templates`
- `links`
- `redirects`
- `langlinks`
- `parse`

其中：

- `allpages` 适合做全量页面发现
- `recentchanges` 适合做增量同步
- `revisions` 适合拉原始 wikitext 与版本信息
- `parse` 适合在少量复杂页面上获取模板展开后的渲染结果

## 2. 共享媒体与文件主数据源：`https://terraria.wiki.gg/api.php`

根站 API 应承担：

- 共享文件仓库元数据
- 文件清单
- 文件 URL、SHA1、尺寸、MIME 信息
- 英文文件说明页补充信息

适合承载的模块包括：

- `allimages`
- `imageinfo`
- 文件页 `info`
- 文件页 `revisions`

适用原因：

- 中文站存在 `imagerepository=terraria_en`
- 这意味着不少文件并不真正属于中文站本地仓库
- 如果后续需求要求“文件不漏”，根站 API 必须参与

## 3. HTML 页面来源：`https://terraria.wiki.gg/zh/`

HTML 页面不适合作为主源，只适合：

- 抽样校验页面展示结果
- 补充极少数模板密集页面的最终渲染结构
- 对 API 返回的原始 wikitext 做人工可读验证

不适合作为主抓取面的原因：

- HTML 噪音大，样式和脚本多
- 数据粒度不稳定，后处理成本高
- 页面层流量更容易触发 Cloudflare 风控
- HTML 抓取更容易被前端结构变更影响

---

## 四、为什么不建议“直接爬页面”

如果主抓取面放在 `https://terraria.wiki.gg/zh/` 页面 HTML，会有几个明显问题。

### 1. 更容易碰风控

实测响应头包含：

- `Set-Cookie: __cf_bm=...`
- `CF-Cache-Status`
- Cloudflare 相关字段

这意味着：

- 页面请求明显经过 Cloudflare 风控体系
- 扫描大量页面 HTML，比调用结构化 API 更像机器人访问
- 在高频、并发、长时间持续访问下，更容易被挑战或阻断

### 2. 页面内容不等于完整结构化数据

HTML 页面虽然能看到展示结果，但不天然适合作为数据库级采集源。缺点包括：

- 页面正文、版本、分类、模板、跳转关系需要二次反推
- 某些语义存在于 wikitext 和模板参数中，不容易从最终 HTML 无损还原
- 某些数据在渲染中会丢失上下文

### 3. 不利于增量同步

HTML 页面天然缺少高效的增量机制。

相比之下，API 有明确的：

- 页面 ID
- 版本 ID
- 时间戳
- 最近变更流
- 分页 continuation

这些能力决定了 API 更适合长期稳定同步。

---

## 五、为什么 API 是主方案

## 1. API 具备全量枚举能力

中文站 API 已确认支持：

- `allpages`

这意味着可以通过标准分页机制完整发现页面，而不是依靠目录页、分类页或站内搜索推断页面范围。

这对“不漏数据”非常关键。

## 2. API 具备增量同步能力

中文站 API 已确认支持：

- `recentchanges`

这意味着后续同步不需要重复全量扫站，只要记录上次同步水位即可追踪新增、编辑、分类变更、日志事件等。

这对降低风控和长期稳定运行都很重要。

## 3. API 具备版本可追踪能力

中文站 API 已确认支持：

- `revisions`

可以获取：

- `pageid`
- `lastrevid`
- `timestamp`
- 原始 `wikitext`
- 页面长度
- 内容模型

这使得数据采集具备：

- 可重放
- 可校验
- 可增量
- 可去重

## 4. API 具备服务端节流信号

实测 `maxlag=1` 请求时，接口返回了：

- `mediawiki-api-error: maxlag`
- `retry-after: 5`
- `x-database-lag: 3540`

这很关键，因为它说明：

- 站点提供了标准的服务端降载信号
- 采集侧可以显式尊重站点负载
- 比盲目扫 HTML 更容易做“温和采集”

因此，从工程可控性看，API 比 HTML 更适合作为正式采集面。

## 六、最推荐的采集策略

## 方案原则

推荐方案遵循四个原则：

- 全量与增量分离
- 正文与媒体分离
- 原始内容与渲染结果分离
- API 主导，HTML 辅助

## 1. 首次全量采集

首次全量建议分三步。

### 第一步：枚举中文页面全集

使用中文 API 枚举页面全集，建立页面主索引。

建议至少覆盖：

- 主命名空间
- 分类
- 模板
- 模块
- 文件
- 指南
- 其他业务上需要的命名空间

如果只关心百科正文，可优先抓内容相关命名空间；如果目标是“Wiki 数据完整归档”，则命名空间不能只抓 `ns=0`。

### 第二步：批量获取页面原始内容与页面关系

对页面主索引逐步拉取：

- 页面基础信息
- 最新修订版本
- 原始 wikitext
- 分类
- 模板
- 内链
- 重定向
- 语言映射

这里应把 `pageid + revid` 视为核心版本标识。

### 第三步：补共享文件与媒体

对中文页面引用到的文件：

- 若文件归属本地中文仓库，则记中文站元数据
- 若文件归属 `terraria_en`，则从根站 API 补全其文件元数据

这是避免图片层漏数的关键步骤。

## 2. 后续增量同步

后续长期运行不应重复全量扫站，而应按变更流同步。

推荐增量面：

- 中文正文变更：`zh/api.php` 的 `recentchanges`
- 文件仓库变更：根站 API 的文件相关增量面

同步逻辑上应保存：

- 最近成功同步时间
- 最近成功同步的变更 continuation
- 最近落库的页面 `lastrevid`
- 最近落库的文件指纹

如果只保存“同步日期”，不保存精确 continuation 或版本水位，长期运行时更容易漏边界变更。

## 七、怎样尽量避免漏数据

“不漏数据”不能靠单次跑通，而要靠机制设计。

## 1. 不能只抓页面正文

只抓页面 HTML 或只抓页面正文会漏掉：

- 分类关系
- 模板依赖
- 文件引用
- 重定向
- 跨语言映射
- 页面版本信息

因此必须把“内容”和“关系”都纳入采集范围。

## 2. 不能只抓中文站文件页

中文站有共享媒体来自英文仓库的情况。  
如果只看中文侧文件页，会出现：

- 文件存在但本地页缺失
- 能拿到 URL 但拿不到完整仓库信息
- 媒体说明页和文件归属不完整

所以文件层必须做仓库归属判断。

## 3. 不能只跑一次全量

Wiki 是持续更新的。  
只做一次全量，之后数据一定会逐步过期。

因此至少需要：

- 全量初始化
- 最近变更增量
- 周期性抽样对账

## 4. 不能忽略 continuation

MediaWiki API 的完整性很大程度依赖 continuation 分页。  
如果中途断了、跳页了、或只取第一页，就一定漏数。

因此 continuation 必须被视为正式同步状态的一部分。

## 5. 需要双向校验

至少要做三类校验：

- 页面总量校验
- 页面版本校验
- 文件引用闭环校验

否则表面“抓到了很多数据”，实际可能已经漏掉页面、漏掉文件或漏掉更新。

---

## 八、怎样尽量避免被风控

“不被风控”本质上不是零风险，而是降低触发概率。

## 1. API 优先，页面次之

这是最重要的一条。

原因：

- API 是结构化接口
- 更容易低频、稳定、可恢复地访问
- 更容易根据 `maxlag`、`retry-after` 做退避
- 比大规模扫 HTML 更接近正常系统对系统访问

## 2. 全量只做一次，长期靠增量

如果每次都全量重扫，会显著提高：

- 请求量
- 连续访问时长
- 重复命中频率

这会明显增加风控概率。

正确策略应是：

- 首次全量
- 后续按变更流增量
- 定期小范围校验

## 3. 并发要保守

站点是否允许高并发并没有公开承诺。  
因此最稳妥的需求设计应默认：

- 并发保守
- 有节流
- 有退避
- 有重试上限

不要把站点当作内部 API 做压力式采集。

## 4. 显式尊重服务端负载信号

既然站点返回了：

- `maxlag`
- `retry-after`

那合理方案就应把它们视为正式协议的一部分，而不是异常噪音。

否则即使短期跑通，长期也更容易被限流或阻断。

## 5. 减少重复访问渲染层

对于没有 `revid` 变化的页面：

- 没必要重复取 HTML
- 没必要重复做 parse

渲染层请求应尽量少，因为它比原始修订层更重。

## 九、推荐的数据采集范围

如果最终要沉淀为 Markdown、知识库、搜索索引或结构化数据库，建议采集范围至少包括以下字段。

## 1. 页面主表

- `wiki_id`
- `pageid`
- `ns`
- `title`
- `canonical_url`
- `is_redirect`
- `latest_revid`
- `touched`
- `content_model`
- `page_language`

## 2. 页面版本表

- `pageid`
- `revid`
- `parentid`
- `timestamp`
- `content`
- `content_format`
- `content_model`

## 3. 页面关系表

- 页面分类
- 页面模板引用
- 页面内部链接
- 页面重定向来源
- 页面跨语言映射

## 4. 文件表

- 文件标题
- 文件归属仓库
- 文件 URL
- 文件说明页 URL
- SHA1
- MIME
- 尺寸
- 上传时间

## 5. 渲染补充表

只对少量复杂页保留：

- parse 后 HTML
- 信息框片段
- 关键表格片段

不建议把所有页面的完整 HTML 当主存储。

## 十、推荐的需求表述方式

如果这个需求后续要给实现方、爬虫方或数据工程方，建议把需求表述成下面这种形式。

## 需求目标

从 Terraria Wiki 中文站及其共享媒体仓库中获取完整且可持续更新的数据，用于后续 Markdown 化、知识库构建或搜索索引，不依赖本地已有文件，不以页面 HTML 为主源，尽量降低站点风控风险。

## 数据源约束

- 中文页面内容主源：`https://terraria.wiki.gg/zh/api.php`
- 共享文件仓库主源：`https://terraria.wiki.gg/api.php`
- HTML 页仅用于少量校验：`https://terraria.wiki.gg/zh/`

## 质量目标

- 页面全集可枚举
- 页面版本可追踪
- 文件引用可闭环
- 增量同步可恢复
- 服务端节流信号被尊重

## 非目标

- 不以浏览器渲染抓取为主方案
- 不依赖人工点击导出
- 不以一次性全量抓取替代长期同步

## 十一、最终推荐方案

综合完整性、稳定性和风控风险，最推荐的正式方案如下：

### 主方案

- 用 `zh/api.php` 做中文页面全量和增量同步
- 用根站 `api.php` 做共享文件仓库补全
- 用 `/zh/` 页面 HTML 做抽样校验和少量复杂页补充

### 为什么这是最合理的

- 完整性最好：页面、修订、关系、文件都能覆盖
- 可持续最好：支持 `recentchanges` 增量
- 风控风险最低：主走 API，不主扫 HTML
- 可校验性最好：有页面 ID、版本 ID、时间戳、文件哈希

### 不推荐方案

不推荐以下方案作为正式主方案：

- 只扫 `https://terraria.wiki.gg/zh/` 页面 HTML
- 只用根站 `api.php` 试图覆盖中文内容
- 每次全量重扫，不做增量同步
- 只抓正文，不抓页面关系和文件层

---

## 十二、验收标准建议

如果要判断方案是否达标，建议至少满足以下验收条件。

### 1. 页面完整性

- 中文站目标命名空间的页面可完整枚举
- 抓取总量与页面枚举总量一致

### 2. 版本完整性

- 每个页面都能拿到最新 `revid`
- 增量同步后，库中 `revid` 与线上最新版本一致

### 3. 文件完整性

- 页面引用文件都能解析到实际 URL
- 共享媒体仓库文件不会因为只抓中文站而漏掉

### 4. 稳定性

- 请求过程支持 continuation 恢复
- 服务端返回 `maxlag` 或 `retry-after` 时能正确退避

### 5. 可维护性

- 后续同步不需要重复全量
- 数据源职责明确：中文内容与共享文件分离

---

## 十三、结论

从你给的两个入口出发，最合理的数据获取方案不是“直接爬中文页面”，而是：

- `zh/api.php` 负责中文页面内容与变更流
- 根站 `api.php` 负责共享媒体仓库补全
- `/zh/` HTML 仅负责抽样校验和少量渲染补充

这套方案最符合以下目标：

- 尽量不漏数据
- 尽量不被风控
- 后续可持续同步
- 可用于整理成 Markdown、知识库或结构化数据

如果后续还要继续细化，可以在此基础上再拆成两份文档：

- 《数据字段清单》
- 《同步与校验策略说明》

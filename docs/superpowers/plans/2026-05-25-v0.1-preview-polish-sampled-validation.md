# V0.1 Preview Polish And Sampled Validation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Keep TerraPedia V0.1 as a local/public preview, polish the visible product surface, and validate data/runtime health with bounded sampled checks instead of full crawls.

**Architecture:** This plan separates release management, visible UI polish, sampled crawler validation, and warning-debt review. It does not run full item-page crawls, imports, backfills, DB writes, or production deployment. Each task ends with a read-only or test-based evidence checkpoint before the next task continues.

**Tech Stack:** Git, Bash local stack scripts, Nuxt front/admin apps, Spring Boot backend, Node test runner, TerraPedia Domain Acceptance reports, bounded wiki item-page sampled smoke.

---

## Scope And Boundaries

### In Scope

- Keep `main` as the active local development base.
- Push the existing local-stack/image-preview fix only when the operator explicitly asks.
- Continue V0.1 preview polish, not formal release.
- Use sampled crawler smoke only, capped at 100 items.
- Triage the 18 A-grade warning panels by product visibility and evidence value.
- Prioritize user-visible issues: broken images, empty Chinese names, bad detail-page data, navigation gaps, obvious mobile/layout defects.
- Produce review notes before merging policy-changing warning cleanup.

### Out Of Scope

- Full item-page crawl or any 3-day crawl.
- `--allow-full-corpus=true`.
- DB import, backfill, apply, load, storage sync, or refresh apply.
- Production deployment, staging promotion, tag creation, or release announcement.
- Cleanup of old merged branches or old worktrees unless separately requested.
- Treating sampled smoke evidence as Domain Acceptance evidence.

### Hard Safety Rules

- This document is a plan and review handoff, not an execution log. Do not treat the sample crawl commands or merge commands below as already run.
- Run command blocks from the repository root unless the command explicitly uses a subshell such as `(cd front-nuxt && ...)`.
- If required local standardized datasets are missing, stop and repair the data-source preflight before any sampled crawler or NPC coverage task. Do not infer missing files from Windows paths.
- Sample reports must not be written under `reports/domain`.
- Domain Acceptance status must continue to come from gate-consumed durable evidence, not ad-hoc UI checks.
- Any discovered data gap that needs DB writes gets a separate plan before execution.
- Any crawler run must include progress visibility and a bounded sample size.

```js
import fs from 'node:fs'

const options = parseArgs(process.argv.slice(2))
const input = options.input || `${process.env.TERRAPEDIA_STANDARDIZED_DIR || 'data/standardized'}/npcs.standardized.json`
const minCoverage = Number(options['min-coverage'] ?? 0.98)

const payload = JSON.parse(fs.readFileSync(input, 'utf8'))
const records = Array.isArray(payload.records) ? payload.records : Array.isArray(payload.npcs) ? payload.npcs : []
const active = records.filter((record) => Number(record.status ?? 1) !== 0)
const withZh = active.filter((record) => String(record.nameZh ?? record.name_zh ?? '').trim())
const coverage = active.length === 0 ? 1 : withZh.length / active.length

const summary = {
  input,
  activeCount: active.length,
  zhNameCount: withZh.length,
  missingZhNameCount: active.length - withZh.length,
  coverage,
  minCoverage
}

console.log(JSON.stringify(summary, null, 2))

if (coverage < minCoverage) {
  console.error(`NPC zh-name coverage ${coverage} is below required ${minCoverage}`)
  process.exitCode = 1
}

function parseArgs(args) {
  const result = {}
  for (const arg of args) {
    const match = /^--([^=]+)=(.*)$/.exec(arg)
    if (match) result[match[1]] = match[2]
  }
  return result
}
```

# TerraPedia 公共前台风格统一设计稿

- Date: `2026-04-11`
- Scope: `front`
- Status: `draft-for-review`
- 设计基线: `HomePage / Atlas Gate`
- 色彩系统: `Moss Lantern`
- 统一方向: `Tiered System`

## 1. 背景与目标

本次设计不是重新做一套全新网站，而是基于首页已经建立的 `Atlas Gate + Moss Lantern` 视觉语言，对公开内容页进行风格统一，让 TerraPedia 从“首页像一个产品、内页像几套页面拼在一起”，收敛成“同一产品家族下的不同页面层级”。

本次设计成功的标志是：

- 首页继续保留最强的品牌和舞台感。
- 公开内页全部回到内容优先、检索优先，不复制首页的重戏剧结构。
- 阅读型页面比检索型页面更有版式节奏，但仍然属于同一设计系统。
- 导航、页头、按钮、卡片、空态、错误态、分页、路径导航不再各做各的。
- 后续实现时，不需要再逐页重新发明视觉逻辑。

## 2. 本次范围

仅覆盖公开内容页：

- `/items`
- `/items/:id`
- `/npcs`
- `/npcs/:id`
- `/articles`
- `/articles/:id`
- `/about`

## 3. 不在本次范围

以下内容明确不在本批次：

- 账号页：登录、注册、找回密码、个人中心
- 后台管理端
- 首页信息结构重写
- 后端接口契约大改
- SEO、部署、内容运营体系重做
- 新增复杂推荐系统或实体关系新接口

说明：

- 首页只作为风格基线和统一源头，不作为本次主要改造对象。
- 如内页改造中发现个别组件必须回改首页公共样式或 token，可做最小必要联动，但不改变首页叙事结构。

## 4. 当前状态诊断

### 4.1 首页现状

首页 `HomePage.vue` 已经建立较完整的品牌语言：

- 明确的 `Atlas Gate` 叙事
- `Moss Lantern` 配色
- 品牌化 Navbar / Footer
- 统一的 surface panel / card 质感
- 探索入口优先的 Hero
- 较完整的明暗主题切换语言

首页的问题不是方向不对，而是它已经走到“产品级壳层”，但大部分公开内页还没有真正跟上。

### 4.2 公开内页现状

当前公开内页大致分成三类：

1. 旧工具型页面
   - 典型代表：`/items`、`/items/:id`
   - 问题：大量旧式 Tailwind 工具类结构仍然主导布局，虽然颜色 token 已接入，但骨架、层次、页头、分页、侧栏仍是旧风格。

2. 半完成的新 token 页面
   - 典型代表：`/npcs`、`/npcs/:id`
   - 问题：已经部分使用 `page-wrap`、`surface-panel`、`section-title` 等系统语言，但尚未与首页形成同等级的一致性，模块之间更像“新卡片套旧结构”。

3. 阅读页与项目说明页
   - 典型代表：`/articles`、`/articles/:id`、`/about`
   - 问题：视觉上比纯工具页更平稳，但仍未形成真正的 Editorial 模板。`Article Detail` 和 `About` 尤其偏轻，缺少足够的阅读节奏与项目表达层次。

### 4.3 核心问题总结

- 视觉系统只在首页完整成立，内页没有统一继承策略。
- 页面角色没有被分层：首页、列表页、详情页、阅读页的表达强度边界不清楚。
- 公共组件语言没有真正落地成“全站母系统”。
- 部分内页仍然依赖旧式结构类名和零散视觉判断。

## 5. 最终采用方向：Tiered System

本次统一方向采用 `Tiered System`，其核心不是“把所有页面做得像首页”，而是把首页的语言拆成可伸缩的层级体系。

### 5.1 层级定义

#### Tier 0 / Atlas Gate

仅保留给首页 `/`。

特征：

- 最强舞台感
- 最强探索入口感
- 双栏 Hero 与装置感视觉
- 品牌与氛围承担第一印象职责

#### Tier 1 / Atlas Workbench

用于检索和实体详情页：

- `/items`
- `/items/:id`
- `/npcs`
- `/npcs/:id`

特征：

- 内容优先
- 检索优先
- 结构比氛围更重要
- 继承首页材质与品牌，但不复制首页戏剧化布局

#### Tier 2 / Editorial Ledger

用于阅读与说明页：

- `/articles`
- `/articles/:id`
- `/about`

特征：

- 比 Workbench 更有阅读节奏和版式呼吸
- 仍然克制，不走首页重舞台
- 更像“冒险记录册 / 编目文档 / 项目手册”

### 5.2 统一原则

- 首页永远是全站表达最强的一页。
- 列表页和详情页优先服务检索任务，不为情绪牺牲效率。
- 阅读页允许多一点叙事性，但必须共享同一套色彩、材质、导航、按钮和状态语言。
- “同产品家族”优先于“每页独特风格”。

## 6. 跨页面设计系统

## 6.1 色彩规则

继续使用首页已落地的 `Moss Lantern` token：

- `--bg-primary`
- `--bg-secondary`
- `--bg-tertiary`
- `--surface-panel`
- `--surface-soft`
- `--surface-elevated`
- `--text-primary`
- `--text-secondary`
- `--text-muted`
- `--border-color`
- `--border-strong`
- `--accent-primary`
- `--accent-secondary`
- `--accent-gold`
- `--accent-support`

使用规则：

- 主 CTA 使用绿色系，不回到蓝色产品按钮。
- 金色只做强调，不做大面积功能色。
- 矿物青只做辅助强调与局部结构点亮。
- 大面积背景维持暖纸色 / 深苔色的低对比层次。
- 不允许页面私自重新引入新的主色体系。

## 6.2 材质系统

公开页只允许三种主表面层级：

1. `surface-panel`
   - 页面级容器
   - 大 Hero、工具条外壳、正文主区

2. `surface-card`
   - 标准内容卡
   - 列表项、详情块、阅读卡、统计块

3. `surface-card--soft`
   - 次级容器
   - 筛选项、标签组、辅助提示、轻量元信息区

不再允许：

- 每页自己定义主容器底色
- 大量直接使用临时 `bg-gray-*` / `border-*`
- 页面级主结构由散乱 utility 背景拼出来

## 6.3 排版系统

统一规则：

- 一级页头使用 `section-eyebrow + section-title + section-copy`
- `section-title` 负责主标题，不再每页自写一套 `h1` 风格
- 检索页正文宽度偏宽，阅读页正文宽度收窄
- 元信息、ID、内部名使用更安静的 label / mono 风格
- 中文为主标题语言，英文名或 internal name 只做 secondary 信息

## 6.4 图标与视觉信号

- 禁止使用 emoji 作为结构性图标
- 优先使用：
  - 字母缩写徽记
  - 低调的内联 SVG
  - 统一 badge / chip 体系
- 图片缺失时统一 fallback 形态，不允许各页各自设计缺图样式

## 6.5 动效规则

内页动效必须比首页弱：

- hover / pressed 只允许轻微位移、边框、阴影变化
- 模块切换优先 opacity / transform，不做大幅位移动画
- 尊重 `prefers-reduced-motion`
- 不允许列表页或阅读页出现首页级漂浮装置感动画

## 6.6 状态系统

以下状态必须统一视觉语言：

- loading
- empty
- error
- retry
- pagination current / disabled
- active filter
- breadcrumb current node

统一要求：

- 状态文案简洁
- 状态卡使用统一 panel 语法
- 不允许每页自己发明一套错误区与空态区

## 6.7 页面骨架

所有公开页共享：

- 同一套 Navbar 壳层
- 同一套 Footer 壳层
- 同一套 `page-wrap` / `page-wrap--narrow`
- 同一套 section spacing 与圆角等级
- 同一套按钮层级

## 7. 页面模板

## 7.1 Atlas Workbench 模板

适用：

- `/items`
- `/items/:id`
- `/npcs`
- `/npcs/:id`

固定结构：

1. 顶部路径层
2. 浅 Hero / 当前页头
3. 工具条或摘要层
4. 主内容区
5. 关联模块区
6. 页尾继续探索层

特点：

- 首屏先解决“我现在在哪、我要查什么、下一步做什么”
- 装饰强度低于首页
- 模块分层明确、信息密度高但不压迫

## 7.2 Editorial Ledger 模板

适用：

- `/articles`
- `/articles/:id`
- `/about`

固定结构：

1. 引导头部
2. 摘要 / 元信息层
3. 主体阅读区
4. 延伸阅读 / 项目补充区

特点：

- 强调阅读节奏与段落层次
- 通过留白、封面、摘要、目录、专题块建立表达
- 仍然共享同一材质和品牌语气

## 7.3 模块密度规则

统一分三档：

- `Dense`
  - 适用：筛选条、属性列表、关系清单、分页
- `Balanced`
  - 适用：标准卡片、详情块、文章卡
- `Airy`
  - 适用：阅读页页头、About 引导区、长文本区

这三档只决定信息密度，不决定是否更换风格系统。

## 8. 逐页设计方案

## 8.1 首页 `/`

本页不作为本次主要改造对象，但它承担风格母版职责：

- 继续保留最强的 Atlas Gate 舞台感
- 继续作为品牌和探索入口
- 内页全部向它继承色彩、材质、导航与品牌语气
- 不从首页复制大场景视觉到内页

## 8.2 物品列表 `/items`

### 页面定位

这是 TerraPedia 的主检索工作台，不是营销页，也不是普通目录页。

### 推荐结构

1. 浅 Hero
   - 左：`Item Index` 标题 + 一句当前页用途说明
   - 右：当前索引状态卡
     - 物品总数
     - 当前分类
     - 当前排序
     - 当前页码

2. Workbench Toolbar
   - 搜索框作为第一主操作
   - 排序器、刷新、当前筛选标签放入同一工具条
   - 工具条整体放入统一 `surface-card--soft` 容器

3. Atlas Index Rail
   - 桌面端保留分类树侧栏
   - 移动端保留抽屉
   - 分类节点统一为 Atlas 索引样式
   - 当前分类要有清晰 active 状态和路径感

4. 结果区
   - 顶部先展示“当前结果摘要条”
   - 中部展示物品卡网格
   - 底部分页单独作为导航层处理

5. 页尾继续探索区
   - 高热根分类
   - 或回到首页探索入口

### 视觉规则

- 不允许继续沿用旧工具页的灰底 + 紧贴布局作为主骨架
- 保留高效率信息密度
- 弱化多余装饰，强化结构清晰度

### 明确不做

- 不将 `/items` 做成首页式双栏舞台
- 不为分类树增加多余装饰插画
- 不把分页和工具栏分散到页面各处

## 8.3 物品详情 `/items/:id`

### 页面定位

这是 Workbench 模板下的标准实体详情页，后续其他实体详情应尽量向它靠拢。

### 推荐结构

1. 路径层
   - 细 breadcrumb
   - 统一返回按钮

2. Hero
   - 左：物品图像台座
   - 右：身份信息
     - 稀有度
     - 分类
     - ID
     - 名称 / secondary name
     - 核心摘要卡
     - 四个关键事实卡

3. 主内容双栏
   - 左主栏：
     - Description & Tooltip
     - Detailed Info
     - Recipes
   - 右侧栏：
     - Category Path
     - Aggregate Metadata
     - 图片 / 来源 / 配方统计
     - 关键元字段

4. 配方模块
   - 作为页面核心模块
   - 先总览，再工作站，再 flow，再 recipe 细项

5. 空态 / 异常态
   - 与公共状态系统统一

### 视觉规则

- Hero 比首页弱，但比普通卡片完整
- 配方区层级明显高于普通属性列表
- 右侧栏永远是辅助索引，不抢主阅读流

### 明确不做

- 不把详情页做成信息卡瀑布流
- 不把配方区降级成普通字段卡
- 不在详情页加入首页级装饰结构

## 8.4 NPC 列表 `/npcs`

### 页面定位

这是人物实体目录页，属于 Workbench，但比物品列表更偏“识别人物并进入详情”。

### 推荐结构

1. 浅 Hero
   - `NPC Directory`
   - 一句说明 TerraPedia 的公开 NPC 索引用途
   - 当前切片摘要：
     - 总条数
     - 当前筛选状态
     - 当前页码

2. 工具条
   - 搜索
   - `Town NPC Only`
   - 当前筛选标签

3. Directory Card Grid
   - 卡片比 Item Card 更宽松
   - 结构：
     - 头像
     - 名称
     - secondary / subName
     - badge
     - Category / Game ID

4. 分页层
   - 与 `/items` 统一组件语言

### 视觉规则

- 更有人物档案气质
- 但不进入 Editorial 阅读页风格
- 仍然是检索与定位优先

### 明确不做

- 不照搬物品小卡网格
- 不用单页海报式人物卡替代目录卡
- 不额外增加首页级装饰背景

## 8.5 NPC 详情 `/npcs/:id`

### 页面定位

这是与 Item Detail 同家族的实体详情页，但内容重点是关系模块而不是配方模块。

### 推荐结构

1. 路径层
   - 与 Item Detail 统一 breadcrumb / back 形态

2. Hero
   - 左：portrait pedestal
   - 右：badges、displayName、secondary / subName、basic profile、facts

3. 关系模块区
   - Loot
   - Shop
   - Buffs
   - 每段统一 section frame：
     - 标题
     - 一句说明
     - 数量 badge
     - 列表 / 空态

4. 可选轻量状态条
   - 若实现成本低，可在 Hero 下增加模块可用性条
   - 仅作为增强项，不强制

### 视觉规则

- Hero 结构与 Item Detail 对齐，确保产品家族一致
- 模块更强调关系浏览
- 空态必须与其他页面统一

### 明确不做

- 不把 NPC 详情做成独立的故事型人物页
- 不让模块卡片各自定义样式
- 不用过强人物视觉装饰破坏 Workbench 效率

## 8.6 文章列表 `/articles`

### 页面定位

这是公开阅读入口页，属于 `Editorial Ledger`。它比 Workbench 更有阅读节奏，但仍然必须承担检索职责。

### 推荐结构

1. Editorial Hero
   - 左：文章页标题、说明、副文本
   - 右：搜索面板
   - 保留当前卡片式搜索结构，但版式更像“编目入口”而不是工具条

2. 可选轻量摘要层
   - 已发布文章数
   - 最近更新时间
   - 当前筛选状态
   - 若暂无现成接口数据，可保留为轻量 meta 文本，不强行增加假数据

3. Article Ledger Grid
   - 文章卡统一风格：
     - 封面
     - 发布时间 / 作者
     - 标题
     - 摘要
     - Continue Reading
     - 编号或阅读标记

4. 分页层
   - 与 Workbench 组共享交互语言
   - 但视觉可更克制，更偏阅读索引

### 视觉规则

- 文章卡比实体卡更强调摘要和节奏
- 页头允许更完整的引导说明
- 不做首页式强探索舞台

### 明确不做

- 不加入大 Featured Hero 文章位，除非后续内容量足够
- 不为文章列表引入复杂杂志化拼贴布局

## 8.7 文章详情 `/articles/:id`

### 页面定位

这是标准阅读页，应明显区别于实体详情页，但仍然保留 Atlas 家族感。

### 推荐结构

1. 返回与路径层
   - 回到文章列表
   - 可保留简化 breadcrumb

2. 阅读页头
   - 封面（有则显示）
   - 标题
   - 发布时间 / 作者
   - 摘要

3. 阅读元信息层
   - 当前建议默认做成页头下方轻量 meta strip
   - 如果标题层级足够，可选目录目录块，但默认不强制

4. 正文区
   - 控制最大阅读宽度
   - 强化正文排版：
     - heading hierarchy
     - code block
     - blockquote
     - list
     - image
   - 优先保证阅读舒适度，而不是信息卡片化

5. 结尾延伸区
   - 返回文章列表
   - 查看更多文章
   - 可选挂一个“继续探索 TerraPedia 数据入口”次级链接

### 视觉规则

- 这是本次范围内最“阅读化”的页面
- 应比 Article List 更安静、更窄、更稳定
- 通过版式和留白表达，不通过大量装饰表达

### 明确不做

- 不把正文区塞进过多 panel 嵌套
- 不在正文页引入强工具栏式结构
- 不做首页级重装饰背景

## 8.8 About `/about`

### 页面定位

这不是一页简单说明，而是 TerraPedia 公共前台的项目基线页，属于 `Editorial Ledger` 下的项目档案页。

### 推荐结构

1. 引导头部
   - 项目定位
   - 这一版 TerraPedia 的核心目标
   - 明确“数据平台 / 公共前台 / 内容入口”的身份

2. 项目快照区
   - Items indexed
   - Categories mapped
   - Running apps
   - Milestone
   - 保留当前统计思路，但整体版式升级为更完整的档案面板

3. 项目基线区
   - Scope
   - Stack
   - 数据闭环
   - 当前公开前台包含什么，不包含什么

4. 路线图 / 公开域说明
   - 当前优先域：Items / NPCs / Articles / About
   - 后续实体扩展方向
   - 这部分应保持真实、克制，不写成营销路线图

5. 入口链接区
   - 去 Items
   - 去 NPCs
   - 去 Articles

### 视觉规则

- About 比 Articles 更“项目文档化”
- 允许更多结构分区，但保持 calm editorial 气质
- 不做纯统计板，也不做营销公司介绍页

### 明确不做

- 不维持当前“两个说明卡 + 一组统计”就结束
- 不把 About 做成后台文档页的前台复刻

## 9. 共享组件与重构目标

为防止实现时再次发散，建议把以下内容抽成统一层：

- Public page hero
- Breadcrumb / back row
- State panel
- Summary strip
- Pager
- Directory card
- Entity hero
- Section frame
- Editorial header
- Editorial meta strip
- Fallback media block

原则：

- 先抽“结构语言”，再做逐页改造
- 不接受同一类模块在不同页面写出两套视觉规则

## 13. 验收结论

本次风格统一完成时，公开前台应达到以下状态：

- 首页仍然是 TerraPedia 的品牌入口和氛围锚点。
- `Items / NPCs` 页面像同一套工作台系统。
- `Articles / About` 页面像同一套阅读与档案系统。
- 所有公开页共享同一壳层、同一状态语言、同一材质体系。
- 内页不再显得像旧页面套新颜色，而是真正属于首页所定义的产品家族。

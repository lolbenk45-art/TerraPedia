## 1. 项目管理方法论：轻量版看板

### 1.1 工具选择

- **平台**：GitHub Projects（看板视图）
- **仓库**：TerraPedia 主仓库

### 1.2 看板列定义

| 列名           | 说明             | 进入条件                         |
| -------------- | ---------------- | -------------------------------- |
| **Backlog**    | 待办池           | 所有已创建但未排期的 Issue        |
| **This Week**  | 本周计划         | Sprint 计划会中选入本周的 Issue  |
| **In Progress**| 进行中           | 正在开发/录入的任务              |
| **Review**     | 待审核           | 已完成，待自检或简单测试         |
| **Done**       | 已完成           | 已合并/发布                      |

### 1.3 WIP 限制

| 列            | WIP 限制 | 目的                     |
| ------------- | -------- | ------------------------ |
| In Progress   | **≤ 2**  | 避免上下文切换，保证专注度 |

> 💡 Solo 开发者精力有限，同时推进 2 个以上任务会显著降低效率。

## 3. 迭代周期

### 3.1 Sprint 节奏

| 阶段           | 时间             | 活动                         |
| -------------- | ---------------- | ---------------------------- |
| **Sprint 计划**| 第 1 个周一      | 从 Backlog 选任务，设定目标  |
| **开发**       | 周二～周五 × 2 周| 按计划执行，每日更新看板     |
| **Review**     | 第 2 个周六      | 自测、整理、文档更新         |
| **发布**       | 第 2 个周日      | 部署、打 Tag、写 Release Notes |

### 3.2 每个 Sprint 目标构成

| 类型         | 数量   | 说明                     |
| ------------ | ------ | ------------------------ |
| **主功能**   | 1 个   | 核心功能或重要模块       |
| **小任务**   | 若干   | Bug 修复、小优化、文档   |
| **数据补充** | 持续   | 按计划录入游戏数据       |

### 3.3 Sprint 日历示例

```text
Sprint 1: 3/17 - 3/30
├── 3/17 (周一)  Sprint 计划
├── 3/18-21      开发周 1
├── 3/24-28      开发周 2
├── 3/29 (周六)  Review
└── 3/30 (周日)  发布 v0.1.0
```

## 4. 优先级框架

### 4.1 优先级定义

| 优先级   | 名称   | 定义                         | 响应时间     |
| -------- | ------ | ---------------------------- | ------------ |
| **P0**   | 必须   | 影响核心功能，用户无法使用   | 立即         |
| **P1**   | 重要   | 改善用户体验，新增重要功能   | 当前 Sprint  |
| **P2**   | 一般   | 优化、小改善                 | 下个 Sprint  |
| **P3**   | 低     | Nice to have                 | 有空再做     |

### 4.2 Eisenhower 矩阵辅助判断

|          | 紧急         | 不紧急       |
| -------- | ------------ | ------------ |
| **重要** | P0：立刻做   | P1：排期做   |
| **不重要** | P2：委托/简化 | P3：搁置     |

### 4.3 标签使用

在 GitHub Issues 中使用 `priority:p0`、`priority:p1` 等标签，便于筛选和排序。

## 5. 进度追踪

### 5.1 GitHub Milestones

- 每个版本对应一个 Milestone
- 命名：`v0.1.0`、`v0.2.0`、`v1.0.0`
- 将 Sprint 内 Issue 关联到对应 Milestone

### 5.2 Sprint Burndown

- **指标**：In Progress + This Week 的 Issue 数量
- **目标**：Sprint 结束时归零
- **简化方式**：用 GitHub Projects 的卡片数量直观跟踪

### 5.3 进度看板示例

```text
Sprint 2 进度 (Day 5/10)
├── Done: 3
├── In Progress: 2 (WIP 已满)
├── This Week: 4
└── 预计完成：可达成 ✓
```

## 6. 复盘机制

### 6.1 Sprint 复盘模板

每个 Sprint 结束后，在项目 Wiki 或 `docs/retrospectives/` 下创建复盘文档：

```markdown
# Sprint X 复盘 (YYYY-MM-DD)

## 做得好 ✅
- 
- 

## 做得不好 ❌
- 
- 

## 改进行动 📋
- [ ] 行动项 1
- [ ] 行动项 2

## 数据
- 计划完成：X / 实际完成：Y
- 主功能：完成 / 未完成
```

### 6.2 月度复盘

每月末进行，关注：

| 维度           | 内容                             |
| -------------- | -------------------------------- |
| **数据指标**   | 录入条目数、页面访问量、功能完成度 |
| **里程碑进度** | 与 Roadmap 对比，是否按计划       |
| **精力状态**   | 是否 burnout，节奏是否可持续     |
| **下月重点**   | 1～3 个核心目标                   |

## 7. 精力管理（Solo 专属）

### 7.1 时间投入原则

| 原则         | 建议                             |
| ------------ | -------------------------------- |
| **每日时长** | 固定 2～4 小时，不贪多           |
| **休息日**   | 每周至少 1 天完全不碰项目        |
| **弹性**     | 身体/状态不好时，允许减量或跳过  |

### 7.2 工作类型区分

| 类型           | 示例                         | 建议时段             |
| -------------- | ---------------------------- | -------------------- |
| **创造性工作** | 写代码、架构设计、UI 设计   | 精力充沛时（如早晨） |
| **机械性工作** | 数据录入、修小 bug、文档整理 | 精力一般时           |

### 7.3 避免 Burnout 清单

- [ ] 不连续加班赶进度
- [ ] 主功能卡住时，先做小任务换脑子
- [ ] 数据录入可拆成 15～30 分钟小段
- [ ] 定期回顾：这个节奏能坚持 3 个月吗？

## 8. 数据录入计划

### 8.1 按游戏进度分批

| 阶段               | 内容范围                 | 优先级 |
| ------------------ | ------------------------ | ------ |
| **Pre-Hardmode**   | 初始装备、NPC、Boss、配方 | 高     |
| **Hardmode**       | 困难模式新增内容         | 中     |
| **Post-Moon-Lord** | 月亮领主后内容           | 低     |

### 8.2 每周录入目标

| 类型         | 建议量     | 说明           |
| ------------ | ---------- | -------------- |
| 物品/武器   | 10～20 条  | 可批量         |
| NPC/Boss    | 1～3 条    | 信息量大       |
| 配方        | 5～15 条   | 依赖物品数据   |

> 根据实际节奏调整，数据录入不应挤压主功能开发时间。

### 8.3 录入工具与自动化规划

| 工具/方式         | 用途                               | 优先级 |
| ----------------- | ---------------------------------- | ------ |
| **批量导入脚本**  | 从 JSON/CSV 导入数据库             | P1     |
| **爬虫/解析**     | 从官方 Wiki 获取基础数据（注意版权）| P2     |
| **表单/后台**     | 单条手动录入界面                   | P1     |
| **数据校验**      | 检查必填项、格式、重复             | P1     |

### 8.4 数据录入 Checklist

- [ ] 定义数据 Schema（字段、类型、必填）
- [ ] 建立录入后台或脚本
- [ ] 制定录入规范（命名、单位、来源标注）
- [ ] 定期备份数据

## 附录：快速参考

### 看板流转

```text
Backlog → This Week → In Progress → Review → Done
```

### 优先级速查

- **P0**：崩了，必须修
- **P1**：重要功能，本 Sprint
- **P2**：优化，下 Sprint
- **P3**：有空再说

### 每日自检

1. 看板上的 In Progress 是否 ≤ 2？
2. 今天计划做创造性还是机械性工作？
3. 本周有没有安排休息日？

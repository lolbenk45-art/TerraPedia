# 需求分析

## 文档目标

本文档按 **2026-04-10** 的仓库现状重写，目标不是继续描述纯设想中的百科站，而是明确：

1. 当前项目已经具备哪些能力
2. 当前阶段真正需要补强哪些需求
3. 后续需求应按什么优先级继续推进

本文档以仓库内现有代码、页面入口、接口入口、数据脚本和标准化数据清单为准。

---

## 一、项目范围重定义

### 1.1 当前真实项目范围

TerraPedia 当前覆盖四类能力：

| 范围 | 当前状态 | 说明 |
|------|----------|------|
| 前台玩家站点 | 已落地 | 面向玩家的物品查询、文章、用户能力 |
| 后台管理端 | 已落地 | 面向运营/维护人员的数据管理工作台 |
| 后端服务 | 已落地 | 承载公共查询、管理接口、认证、文章、聚合查询 |
| 数据工程链路 | 已落地 | 抓取、标准化、导入、回填、审计、同步 |

### 1.2 当前不再适合的旧定义

以下表述已经不适合作为主需求基线：

- “项目仍处于脚手架阶段”
- “只有物品百科是核心范围”
- “管理端和数据管道后续再考虑”
- “先做静态站点，再决定要不要后端”

这些内容与当前仓库状态不符。

### 1.3 当前需求主线

当前阶段的核心需求应聚焦为：

1. **把多源游戏数据稳定沉淀为统一数据资产**
2. **让后台能高效维护重点实体和复杂关系**
3. **让前台稳定消费已经治理过的数据**
4. **把内容发布、用户体系、数据治理逐步纳入统一流程**

## 二、主要用户与使用场景

### 2.1 外部用户

| 用户类型 | 当前诉求 | 当前对应能力 |
|------|----------|-------------|
| 新手玩家 | 快速找到物品用途、来源、配方 | 前台首页、物品列表、物品详情、搜索建议 |
| 中期/硬核玩家 | 查看更完整的属性与关系数据 | 物品聚合详情、配方树、来源与图片信息 |
| 内容读者 | 浏览攻略/文章 | 文章列表、文章详情 |
| 注册用户 | 登录、维护个人资料、撰写内容 | 登录注册、找回密码、个人中心、文章写作台 |

### 2.2 内部用户

| 用户类型 | 当前诉求 | 当前对应能力 |
|------|----------|-------------|
| 数据维护人员 | 修正物品、分类、配方、实体数据 | 后台管理端多实体工作区 |
| 运营/编辑人员 | 管理文章与用户 | 文章管理、用户管理 |
| 数据工程人员 | 抓取、导入、审计、回填 | `scripts/data/*` 脚本体系 |
| 开发人员 | 联调前后台与数据库 | 本地启动脚本、验证脚本、SQL 查询页 |

### 2.3 当前阶段核心诉求

1. **数据要可追踪**
   - 数据从哪里来、如何标准化、如何导库，应可解释
2. **后台要可维护**
   - 重点实体不能只能靠脚本改库，必须有可视化维护入口
3. **前台要可消费**
   - 已经治理好的数据应尽快在前台形成用户可见价值
4. **需求要和代码同步**
   - 文档不能继续停留在“未来可能会做”的层面

## 四、当前在建需求

以下需求已经有代码入口或局部实现，但仍应视为当前阶段重点，而不是“以后再说”。

### 4.1 数据治理闭环补强

#### R01 多源数据的可验证导入

需求：

- 抓取结果要有更清晰的 manifest、审计结果和导入前校验
- 标准化结果要能区分可直接导库与待人工确认数据
- 重点 pipeline 需要 dry-run、样本核验和异常收口

原因：

- 当前数据脚本已很多，必须防止“能跑但不可验”

#### R02 重点实体质量校验

需求：

- 配方、Shimmer、Town NPC、Boss/NPC 关系类数据要补充更清晰的测试与样本验收
- 后台编辑结果与数据库结果要能快速复核

原因：

- 这些模块已进入业务维护阶段，质量风险高于静态展示页

### 4.2 后台产品化补强

#### R03 后台工作区统一性

需求：

- 后台各页面在筛选、分页、保存、错误提示、返回链路上保持一致
- 复杂实体维护页提供更明确的“只读信息 / 可编辑信息 / 关联关系”区分

#### R04 批量维护与校验能力

需求：

- 对导入型模块逐步补充批量校验、批量修正、差异提示能力
- 避免大部分数据维护都退回到脚本或手工 SQL

### 4.3 前台扩展需求

#### R05 从物品域扩展到更多实体域

需求：

- 将后台已维护的 NPC、Boss、Biome、Buff 等实体，逐步转成前台可访问页面
- 前台实体页应优先复用后端聚合查询能力

#### R06 内容与数据双线并行

需求：

- 文章系统不能只停留在后台或写作台
- 需要明确文章如何与实体词条、专题页、前台导航形成关系

## 六、非功能需求

### 6.1 数据一致性

当前项目的数据一致性优先级高于纯视觉指标。

要求：

- 同一实体在标准化数据、数据库、后台页面、前台页面中的核心字段口径一致
- 导入脚本支持 dry-run 或等价验证模式
- 对高风险数据变更保留审计或快照信息

### 6.2 可维护性

要求：

- 数据脚本按职责分目录维护
- 后端接口按实体域分层
- 管理端页面具备统一导航和工作区结构
- 文档与当前仓库状态保持同步

### 6.3 性能

当前阶段的性能要求应更贴近真实系统：

| 场景 | 当前目标 |
|------|---------|
| 前台常用列表接口 | 常规查询可接受，无明显阻塞 |
| 前台详情聚合接口 | 可在用户可接受时间内完成展示 |
| 后台管理页首屏 | 常用页面可正常加载与操作 |
| 本地联调启动 | 三端可通过脚本完成启动与验证 |

说明：

- 当前文档暂不再写“全球 CDN 延迟 < 500ms”这类与仓库现状脱节的指标
- 生产环境性能目标应在部署方案明确后再细化

### 6.4 安全

要求：

- 管理端与用户端认证分离清晰
- 本地配置与凭据不入库
- 图片上传与对象存储走受控路径
- 管理接口默认受保护

### 6.5 可测试性

要求：

- 后端重点实体接口有最小编译/测试保障
- 管理端至少通过类型检查
- 数据任务具备 dry-run、样本比对或 audit 结果

## 七、验收口径

### 7.1 当前阶段验收重点

当前阶段不以“页面数量”作为主要验收标准，而以闭环为主：

1. 有数据来源
2. 有标准化产物
3. 有导入或同步入口
4. 有后台维护入口
5. 有最小验证方式
6. 有前台消费价值或明确的前台接入计划

### 7.2 不应再采用的旧验收方式

- 只看是否做出页面
- 只看是否能抓到数据
- 只看是否已经部署上线
- 只看是否写了需求文档

这些都不足以说明当前 TerraPedia 项目是否真正推进。

## 九、结论

TerraPedia 当前的核心问题，已经不是“要不要做一个百科站”，而是：

- 如何管理不断增长的结构化游戏数据
- 如何把脚本能力、后台能力、前台能力和内容能力接成一条线
- 如何用更准确的需求文档支撑后续实现与验收

因此，后续需求文档应默认遵循以下原则：

1. 先写清楚当前已存在的能力
2. 再写清楚正在补强的闭环
3. 最后才写远期规划

这样才能与现在的项目阶段匹配。

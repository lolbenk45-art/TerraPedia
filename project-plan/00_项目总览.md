# 项目总览

## 项目定义

**TerraPedia** 不再只是一个“泰拉瑞亚中文百科站”规划稿，而是一个已经进入实现阶段的游戏数据平台项目。

截至 **2026-04-10**，仓库内已经形成四条并行主线：

1. **前台玩家站点**
   - Vue 3 + Vite
   - 已有首页、物品列表、物品详情、文章列表/详情、文章写作台、用户登录注册与个人中心
2. **后台管理端**
   - Nuxt 4
   - 已有物品、分类、配方、制作站、任意物品组、Shimmer、NPC/Boss/Buff/Projectile/Biome、世界条件、文章、用户、数据查询等工作区
3. **后端服务**
   - Spring Boot + MyBatis Plus + Flyway + Redis/Redisson + MinIO
   - 已提供公共查询接口、管理端接口、认证接口、文章接口、聚合查询接口
4. **数据抓取与同步链路**
   - 已有抓取、标准化、导入、回填、审计、同步、pipeline 编排脚本
   - 已形成 wiki 数据接入与本地数据库维护能力

项目的真实定位应更新为：

> 一个以结构化数据为底座，覆盖“抓取同步、后台维护、前台查询、内容发布”的 Terraria 中文数据平台。

## 当前阶段判断

### 已完成阶段

**基础底座阶段**

- 前后端项目已拆分并可独立开发
- 后端数据库迁移机制已建立
- 数据脚本目录与分层职责已成型
- 前台物品检索与详情主链路已可开发和验证
- 后台管理端已不是空壳，而是覆盖多个实体域

### 当前所在阶段

**数据治理与运营后台强化阶段**

当前最核心的工作，不是再写一版理想化需求，而是继续补强以下闭环：

1. 数据抓取结果稳定进入标准化层
2. 标准化数据稳定导入数据库
3. 管理端能对重点实体进行人工校正
4. 前台能消费聚合后的稳定数据
5. 内容系统与用户系统逐步纳入运营流程

### 下一阶段

**前台内容扩展与发布质量阶段**

- 从“物品为主”扩展到 NPC、Boss、Biome 等前台可访问页面
- 强化 SEO、结构化元信息与公开可访问体验
- 完善文章发布链路与内容消费体验
- 将更多后台维护结果直接反映到前台用户可见页面

## 当前成功指标

现阶段不宜继续使用“日均 UV”“月均 PV”作为唯一判断标准，应该先以工程完成度为主。

### 工程指标

| 指标 | 当前目标 |
|------|---------|
| 本地三端启动 | 可通过脚本完成 |
| 前台物品主链路 | 列表、详情、搜索、分类可联调 |
| 后台重点域 | 至少覆盖物品、分类、配方、实体、文章、用户 |
| 数据导入链路 | 抓取、标准化、导入、审计具备可执行脚本 |
| 数据规模 | 维持现有标准化数据集并持续增量更新 |

### 产品指标

| 指标 | 当前目标 |
|------|---------|
| 前台公开能力 | 先稳定物品域，再扩展其他实体域 |
| 数据一致性 | 后台维护结果与前台展示口径一致 |
| 文档一致性 | 需求文档、项目总览、代码状态尽量同频 |

## 主要风险

| 风险 | 当前影响 | 说明 | 应对策略 |
|------|----------|------|----------|
| 文档仍停留在规划期 | 高 | 会误导后续排期与验收 | 用当前代码状态重写需求与阶段定义 |
| 数据链路复杂度持续上升 | 高 | 抓取、标准化、导库、管理端之间耦合增加 | 强化分层与脚本入口规范 |
| 管理端覆盖范围扩张过快 | 中 | 页面多但校验和测试可能不足 | 先补关键域测试和验收清单 |
| 前后台能力不对齐 | 高 | 后台能维护但前台未消费，或前台需求缺少后台支撑 | 以聚合查询接口作为前后台对齐中轴 |
| 多实体数据质量波动 | 中 | NPC/Boss/Shimmer/配方等数据来源复杂 | 通过 audit、dry-run、样本校验降低风险 |

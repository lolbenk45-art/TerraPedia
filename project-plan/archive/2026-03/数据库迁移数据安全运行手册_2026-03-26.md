# TerraPedia 数据库迁移数据安全运行手册（2026-03-26）

## 1. 目标

本手册用于约束后续数据库重构过程中的数据安全、可追溯性和回退能力。

原则只有三条：

- 不允许在未备份的情况下执行任何 DDL 或回填脚本
- 不允许在当前活跃数据库上直接做首次验证
- 不允许让新表在第一阶段成为唯一真源

## 3. 强制流程

每次数据库动作必须按以下顺序执行：

1. 记录当前 Git 状态
- 当前分支
- 当前提交
- 当前 worktree 路径

2. 创建数据库备份
- 结构备份
- 全量数据备份
- `items`
- `category`
- `flyway_schema_history`

3. 生成备份清单
- 备份文件路径
- 文件哈希
- 导出时间
- 操作人
- 目标数据库名
- 对应 Git 提交

4. 在影子数据库先演练
- 不直接对当前活跃库执行首轮迁移
- 首轮验证数据库必须是新库名或临时恢复库

5. 执行迁移
- 只允许新增迁移版本
- 不允许修改历史已执行迁移

6. 执行回填
- 配方、来源关系、biome、图片元数据回填不进入 Flyway
- 统一通过独立脚本执行

7. 执行校验
- 表结构存在性
- 关键表行数
- 旧接口可用性
- 新接口只读验证

8. 通过后再考虑切换读路径
- 第一阶段不删除旧字段
- 第一阶段不删除旧表

## 4. 必须保留的备份内容

最低要求：

- 完整 schema 备份
- 完整库全量备份
- `items` 单表备份
- `category` 单表备份
- `flyway_schema_history` 单表备份
- 备份 manifest
- 备份文件哈希

推荐额外保留：

- 应用配置快照
- 本次迁移 SQL 文件副本
- 本次回填脚本副本
- 本次校验结果

## 5. 可追溯性要求

每次迁移或回填都必须能追溯到：

- 谁执行的
- 在哪个时间执行的
- 针对哪个数据库执行的
- 基于哪个 Git 提交执行的
- 使用了哪份 SQL / 脚本
- 使用了哪份备份
- 对应哪次校验结果

建议每次执行都生成：

- `manifest.json`
- `checksums.txt`
- `row-counts.tsv`
- `git-context.txt`

## 6. 回退原则

### 6.1 结构回退

- 第一阶段全部采用 additive migration
- 新表创建失败或读路径异常时，优先回退应用读逻辑
- 非必要不立即删除新表

### 6.2 数据回退

如果出现数据异常，回退顺序如下：

1. 停止回填脚本
2. 切回旧应用读路径
3. 核对 `items/category/flyway_schema_history` 备份
4. 必要时恢复到迁移前备份库

### 6.3 清理原则

以下动作至少延后一个稳定版本：

- 删除旧字段
- 删除旧表
- 让新表成为唯一真源

## 7. 多 Agent 协同约束

为了避免多人并行导致的数据丢失或方案漂移，必须遵守：

- 所有 schema 变化先进入设计文档再进入 SQL
- 所有迁移文件命名和顺序统一由主控 agent 收敛
- 不允许不同 agent 在同一张迁移文件上并行写入
- 回填脚本与 DDL 脚本分离
- 所有落地动作都必须带 Git 检查点

## 8. 第一阶段禁止事项

- 禁止直接重写 `items` 主表为全新模型
- 禁止直接把当前活跃数据库作为首轮验证环境
- 禁止修改历史 Flyway 迁移
- 禁止不做备份直接跑 DDL
- 禁止先删旧字段再切新逻辑

## 2. 当前状态判断

### 2.1 隔离分支基线

隔离分支当前后端基线仍是早期模型：

- `items` 单表
- `category` 树
- `ItemDTO` / `ItemVO` 为平铺结构
- 仅有 `schema.sql`
- 当前提交内没有 `back/src/main/resources/db/migration`

### 2.2 用户新增的数据面

本轮数据库设计需要覆盖：

- item 主数据
- 图片来源与缓存
- 获取方式 / 来源关系
- 合成配方
- biome / 群系
- 数据抓取来源元数据与快照

这些需求均来自：

- `project-plan/03_架构设计.md`
- `project-plan/数据来源分析与真实Demo汇总_2026-03-26.md`

## 3. 设计原则

### 3.1 总原则

- 不做“推翻旧表的一次性大改”
- 采用“旧表保留 + 新表并行 + 应用兼容层 + 延迟清理”
- 第一阶段只做加法，不做破坏性替换

### 3.2 规范化边界

建议规范化：

- 一对多图片
- 一对多获取方式
- 一对多配方
- 配方原料
- 配方工作台
- biome 主表及 biome 间关系
- biome 与资源关系

不建议过度规范化：

- `damage / defense / knockback / use_time / width / height / buy / sell`
- 当前阶段不要强行把 `boss / npc / shop / event` 全量外键化
- 当前阶段不要把所有 wiki 长尾字段做成超宽 `items` 表

### 3.3 与历史策略保持一致

建议继续避免物理外键，采用逻辑关联字段和索引。

原因：

- 项目历史上已经处理过物理外键调整
- 当前重构阶段更需要“迁移弹性”和“失败可回退”
- 外部数据回填具有不完整性，逻辑引用更稳

## 4. 推荐数据库模型

最推荐的版本是“V2 混合规范化模型”：

- 保留 `items` 作为核心聚合根
- 保留 `category` 作为现有分类树
- 将图片、来源关系、配方、biome 等拆到独立关系表
- 用快照表承接原始抓取 payload 和重解析需求

## 7. Flyway 与迁移策略

### 7.1 重要前提

本隔离分支当前提交里没有 Flyway 迁移链，因此：

- 这里适合写设计稿和 DDL 草案
- 真正实施时，应把方案迁移到“已有 V1-V11 的最新 Flyway 主线”

### 7.2 迁移原则

- 只新增迁移版本
- 绝不修改已执行过的旧迁移
- Flyway 只负责结构和少量确定性 seed
- 外部 wiki 数据同步、配方回填、掉落回填、biome 回填，不放进 Flyway

### 7.3 推荐迁移拆分

- `V12__create_recipe_and_acquisition_tables.sql`
- `V13__create_biome_and_image_tables.sql`
- `V14__seed_core_biomes.sql`
- `V15__add_item_metadata_columns.sql`

如需回填：

- 独立导入脚本处理，不走 Flyway

## 8. 回退策略

### 8.1 结构回退

- 第一阶段都是 additive migration
- 即使失败，也优先通过应用切回旧读路径恢复
- 不建议立即 `DROP` 新表

### 8.2 数据回退

每次 DDL 前至少保留：

- schema 备份
- `items/category` 全量备份
- `flyway_schema_history` 备份

### 8.3 发布回退

- 新表即使建错，也不影响旧接口继续读旧表
- 旧字段和旧接口必须保留至少一个发布周期

## 先下结论

截至 2026-04-17，**TerraPedia 已经是一个实质进入实现阶段的多模块工程项目**，而不是停留在规划期的百科站概念稿。

- 项目底座已经具备，前台、后台、后端、数据脚本四条主线都不是空壳

- `wiki-crawler-v1` 已经形成了可跑、可测、可产出报告的 NPC 数据闭环雏形

## 项目阶段判断

按照当前项目总览与仓库真实状态，TerraPedia 的阶段判断仍然成立：

### 已完成的部分

- 前台、后台、后端、数据脚本目录结构已经拆开
- 数据抓取、标准化、导入、审计、同步的基础路径已经建立
- 物品域前台主链路已经成型
- 后台管理端已经覆盖多个实体域
- 本地联调脚本已经具备启动、停止、校验入口

### 当前所在阶段

当前仍属于：

**Phase B：数据治理与管理闭环强化阶段**

当前更重要的问题不是继续堆页面，而是继续补强以下闭环：

1. 抓取结果稳定进入标准化层
2. 标准化数据稳定进入数据库或可消费产物
3. 管理端可以对重点实体做人工维护和校正
4. 前台可以消费稳定的聚合结果
5. 各条链路之间具备更明确的验收标准

### 下一阶段的进入条件

只有在 NPC、Boss、Biome 等重点域满足以下条件后，项目才更适合进入前台公开能力扩展阶段：

- 数据源有稳定入口
- 管理端有可操作入口
- 公共接口有可验证输出
- 前台页面能消费同一口径的数据
- 最小测试与最小联调验证具备可重复执行方式

## 当前主要风险

### 1. 主仓库改动面持续扩大

`main` 上当前改动已经覆盖多个模块，若不分主题收口，后续会出现：

- review 范围失控
- 变更因果难以追踪
- 回归验证成本上涨

### 2. 前后台与数据脚本仍可能口径脱节

当前能看到三条线都在推进：

- 后端公共聚合
- 前台 NPC 页面
- 数据脚本与 crawler

如果没有统一的验收标准，很容易形成：

- 管理端能维护
- 脚本能产出
- 前台也有页面
- 但三者不是同一口径

### 3. `wiki-crawler-v1` 仍是“有效样板”，不是“NPC 域完成态”

虽然这条线现在最扎实，但现状仍然是：

- 有闭环
- 有报告
- 有测试
- 但覆盖率仍有限

如果过早把它当成“NPC 已完成”，后续容易在主仓库整合时低估缺口。

### 4. 运行态与代码态仍未完全打通

当前本地三端脚本是存在的，但本次快照没有基于活跃服务实例完成联调判断。

因此目前能确认的是：

- 代码在动
- 报告在产出
- 子系统测试在通过

但还不能据此自动推出：

- 主仓库当前可以无阻塞地完整联调
- NPC 公共链路已经完成端到端验收

## 近期建议

### 建议一：先把 NPC 域做成真正的样板闭环

优先继续推进 `wiki-crawler-v1`，目标不是马上全量跑完，而是：

- 持续按高收益分片补覆盖
- 收掉剩余 warning
- 提升 bridge 富化比例
- 固化 coverage 与 bridge 作为验收报告

理由很简单：当前这条线已经最接近“可重复、可验证、可持续扩”的工程样板。

### 建议二：再打通主仓库 NPC 前后台整合

在 `wiki-crawler-v1` 稳定扩覆盖的同时，主仓库需要重点核对：

1. 后端 NPC 聚合接口口径
2. 管理端 NPC 维护入口口径
3. 前台 NPC 页面消费口径
4. 数据脚本产物接入点口径

目标不是继续各自推进，而是回答一个更硬的问题：

> 主仓库中的 NPC 公开链路，是否已经开始围绕同一份数据事实运转。

### 建议三：对主仓库当前改动按主题分批收口

建议优先按以下主题拆分和整理：

- NPC 公共查询与聚合
- NPC 前台页面与 API
- 管理端 NPC / Town NPC 维护
- 本地栈兼容与启动稳定性
- fallback / cache / request 基础设施

如果不这样做，当前 `115` 个未收口变更项会持续抬高后续决策噪音。

## 最终判断

TerraPedia 现在最真实的状态，不是“快做完了”，也不是“还停在规划期”，而是：

> 项目整体已经有平台级骨架，正在从多条分散能力线向重点域闭环收敛；其中 NPC crawler 是目前最具工程完成度的增量成果，但主仓库仍明显处于高活跃、未收口、待整合的阶段。

如果后续推进顺序正确，当前阶段最值得做的事不是继续无差别扩张，而是：

1. 把 NPC 域做成样板闭环
2. 用样板反推主仓库的验收口径
3. 再决定下一批扩域工作如何进入前台公开阶段

这份快照的价值，不在于证明项目“看起来做了很多”，而在于给后续所有技术判断提供一个更硬的起点。

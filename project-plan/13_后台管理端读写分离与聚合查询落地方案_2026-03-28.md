# TerraPedia 后台管理端读写分离与聚合查询落地方案（V1）

日期：2026-03-28  
状态：待执行（本方案通过后作为后续实现唯一基线）  
适用范围：`terraPedia-db-redesign` 后端 `back`、前端 `data-query-app`、数据导入脚本、测试与发布流程

---

## 1. 背景与决策

### 1.1 当前矛盾

1. 后台管理端需要按模块直观维护数据（Item、Recipe、Source、Biome、Npc、Projectile 等）。
2. 前端读取又需要“以 Item 为中心”的聚合视图，减少多接口拼装和不一致快照。
3. 现有数据库以逻辑关联为主（弱外键/软关联键较多），接口拆分导致读取链路复杂。

### 1.2 核心决策

1. **写链路不聚合**：保留现有后台 CRUD 分模块设计，不做“大一统写接口”。
2. **读链路聚合**：新增 Query Side（轻量 CQRS），只用于前端读取和展示编排。
3. **先代码读写分离，再数据读写分离**：V1 同库聚合；V2 视情况引入读投影表。
4. **可回滚优先**：聚合能力受开关控制，任何异常可快速退回旧接口。

---

## 2. 多角色讨论结论（多 Agent 评审）

本方案由五类角色并行评审后合并：

1. 后端架构师：建议采用轻量 CQRS，新增 `CatalogQueryFacade + ItemAggregateQueryService`，写侧不动。
2. 前端工程师：只改 `stores/items.ts` 适配层，不改页面调用签名，聚合失败自动回退旧接口。
3. UI/UX 工程师：详情页采用“模块总览条 + 模块卡片分区”，空态/错态/加载态标准化。
4. 测试工程师：补齐“单测+集成+契约+回归”矩阵，新增 CI 强阻断项。
5. DevOps 工程师：采用“数据灰度 -> 流量灰度”双阶段发布，明确告警阈值与回滚触发。

---

## 3. 目标与非目标

### 3.1 目标

1. 新增 Item 聚合读取接口，减少前端跨接口拼装。
2. 保持后台管理端各模块可独立查看和编辑数据。
3. 建立前后端、测试、发布一体化落地路径（可执行、可验收、可回滚）。

### 3.2 非目标

1. 不在本期重构所有 `/admin/*` 写接口。
2. 不在本期引入重型事件溯源体系。
3. 不在本期一次性恢复全量物理外键。

---

## 4. 目标架构（V1）

### 4.1 架构分层

1. **Command Side（现有）**
   - `/items`、`/admin/*` 继续负责增删改。
   - 复用现有 Service/Mapper，不改变管理端操作模型。
2. **Query Side（新增）**
   - 新增 `/public/*` 聚合只读接口。
   - 通过 Facade 编排 Item、Image、Recipe、Source、Category 等数据。
3. **Cache & Consistency**
   - 写入后触发缓存失效。
   - 聚合接口先同步刷新，后续可扩展异步投影刷新。

### 4.2 读写边界原则

1. Query Side 严禁直接写库。
2. Command Side 不返回聚合大对象，避免写接口耦合膨胀。
3. Query DTO 与 Command DTO 解耦，避免字段漂移互相污染。

## 5. API 方案（V1 可实施）

### 5.1 新增接口

1. `GET /public/items/{id}/aggregate`
   - 查询参数：`include=images,sources,recipes,category`（可选，默认 `images,sources,recipes,category`）
   - 作用：单个 Item 聚合详情
2. `GET /public/items/aggregate`
   - 查询参数：`page,limit,search,categoryId,rarityId,sortBy,include`
   - 作用：聚合列表（含基础统计字段）
3. `POST /public/items/aggregate/batch`
   - 请求体：`{ "ids":[1,2,3], "include":["images","sources"] }`
   - 作用：批量详情读取
4. `GET /public/categories/tree-with-counts`
   - 查询参数：`rootId`（可选）
   - 作用：分类树 + 计数，前端筛选面板直接消费

### 5.2 响应契约（统一）

```json
{
  "success": true,
  "message": "ok",
  "data": {
    "item": {},
    "images": [],
    "sources": [],
    "recipes": [],
    "category": {},
    "moduleStatus": {
      "images": "ok",
      "sources": "ok",
      "recipes": "empty",
      "category": "ok"
    },
    "meta": {
      "updatedAt": "2026-03-28T12:00:00Z"
    }
  },
  "pagination": {
    "page": 1,
    "limit": 20,
    "total": 100,
    "totalPages": 5
  }
}
```

### 5.3 字段兼容约束

1. 同时兼容 camelCase / snake_case（前端现有 normalize 逻辑依赖）。
2. 列表和详情保持关键字段同名：`id,name,image,status,categoryId,categoryName,rarityId,updatedAt`。
3. include 子模块失败不得使主数据失败，必须通过 `moduleStatus` 暴露局部错误。

## 6. 后端实施设计

### 6.1 复用与新增

1. 复用：
   - `ItemService`
   - `ItemImageService`
   - `ItemSourceService`
   - `RecipeService`
   - `CategoryManagementService`
2. 新增：
   - `CatalogQueryFacade`
   - `ItemAggregateQueryService`
   - `ItemAggregateAssembler`
   - `ItemReadQueryMapper`（批量查询，避免 N+1）
   - DTO：`ItemAggregateDTO`、`ItemListAggregateDTO`、`AggregateIncludeOption`

### 6.2 SQL 与性能改造

1. 复合索引（优先执行）：
   - `item_images(item_id, deleted, status, sort_order, id)`
   - `item_acquisition_sources(item_id, deleted, status, sort_order, id)`
   - `recipes(result_item_id, deleted, status, id)`
   - `recipe_ingredients(recipe_id, sort_order, id)`
   - `recipe_stations(recipe_id, sort_order, id)`
   - `items(deleted, category_id, rarity_id, id)`
2. 优化点：
   - 将 `ItemMapper.xml` 中逐行“取首图”子查询改为派生表或窗口函数聚合后一次 JOIN。
3. 一致性：
   - 统一读路径 `deleted=0` 过滤策略。
   - 增加每日关联巡检任务（孤儿数据、软关联失配、排序异常）。

### 6.3 软约束与数据完整性

1. 不强推全表 FK，但对核心链路增加“写前存在性校验”。
2. 对 `source_item_id` 与 `item_id` 双轨字段建立对账脚本和告警阈值。
3. 聚合读取必须可追踪 `source` 元数据，便于后台排障和人工修正。

## 7. 前端实施设计

### 7.1 改造原则

1. 页面调用签名尽量不变，改造集中在 `stores/items.ts`。
2. 开关优先，支持快速回退到旧接口。
3. 分波次迁移，不做全量一次切换。

### 7.2 迁移波次

1. 波次1：`pages/items.vue` 列表切聚合列表接口。
2. 波次2：`components/ItemDetail.vue` 接聚合详情（先基础+images，再 recipes/sources）。
3. 波次3：`pages/index.vue` 最近物品卡片复用聚合列表。
4. 波次4：`pages/entities/[type].vue` 按类型渐进接入聚合能力（可选）。
5. 波次5：评估 categories/其他后台页是否受益，决定是否扩展。

### 7.3 状态与缓存策略

1. 缓存 Key：
   - 列表：`page+limit+search+categoryId+rarity+include`（TTL 30-60s）
   - 详情：`itemId+include`（TTL 5-10min）
2. include 策略：
   - 列表最小 include
   - 详情模块懒加载
3. 降级策略：
   - 聚合失败自动回退旧接口
   - 子模块失败不阻断主数据渲染

## 8. UI/UX 方案（使用 ui-ux-pro-max 规范）

### 8.1 信息架构

1. 基础信息区：名称、ID、分类、状态、稀有度、核心属性。
2. 关联模块区：图片、配方、来源、群系，模块固定可见。
3. 数据来源区：展示 `sourceType/sourcePage/sourceFileTitle/同步时间/可信级`。

### 8.2 交互规范

1. “模块总览条 + 模块卡片分区”，每块独立加载/错误/空态。
2. 模块默认展开前2个，其余记忆用户上次状态。
3. 保存后显示变更摘要条，批量操作显示成功/失败明细入口。

### 8.3 组件改造包（最小可交付）

1. 新增 `ModuleVisibilityPanel.vue`
2. 新增 `RelationModuleCard.vue`
3. 新增 `ChangeSummaryBar.vue`
4. 改造 `ItemDetail.vue`
5. 改造 `items.vue`（关联完整度列 + 骨架 + 结果面板）
6. 改造 `AppEmptyState.vue`（支持 `variant`）
7. 改造 `AppToast.vue`（action + `aria-live`）

### 8.4 可访问性清单（必须）

1. 全交互可键盘访问（Tab/Enter/Esc）。
2. 焦点样式统一 `:focus-visible`。
3. 状态不只依赖颜色，文本和图标并存。
4. 表格排序补 `aria-sort`，批量勾选有“已选N项”读屏提示。

## 9. 测试与质量门禁

### 9.1 测试矩阵

1. 单测：Service/Controller 聚合逻辑分支覆盖。
2. 集成：MySQL + Redis + Flyway + MockMvc（建议引入 Testcontainers）。

4. 回归：P0 接口烟测 + 历史缺陷回归集。

### 9.2 CI 强阻断

1. 单测失败阻断。
2. 集成测试失败阻断。
3. 契约破坏兼容阻断。
4. Flyway 迁移失败阻断。
5. 前端 typecheck 失败阻断。
6. P0 回归失败阻断。

### 9.3 覆盖率与性能基线

1. 行覆盖率 >= 70%，分支覆盖率 >= 60%。
2. 关键模块（鉴权、图片、搜索、聚合）分支覆盖率 >= 80%。
3. 聚合读取接口 P95 初期目标 < 300ms（预发与生产监控一致口径）。

## 10. 发布、灰度、回滚

### 10.1 发布流水线（固定）

1. `back/scripts/db/backup_and_verify.ps1`
2. `scripts/data/standardize-existing-data.mjs`
3. `scripts/data/import-standardized-to-db.mjs`
4. `scripts/data/import-independent-entities-to-db.mjs`
5. `scripts/data/validate-independent-entity-item-links.mjs`

### 10.2 灰度策略

1. 先数据灰度，再流量灰度。
2. 流量比例：5%（30m）-> 20%（30m）-> 50%（60m）-> 100%。
3. 升档前提：`hitRatePercent >= 99%` 且 `unmatched` 无持续增长。

### 10.3 回滚策略

1. 配置回滚：关闭聚合开关，切回旧读接口。
2. 应用回滚：回退版本并重部署。
3. 数据回滚：使用备份 `full.sql + checksums + manifest` 恢复。

### 10.4 回滚触发阈值

1. 5xx > 2% 持续10分钟。
2. P95 延迟较基线翻倍并持续10分钟。
3. 缓存命中率下降 > 15% 且持续15分钟。
4. `unmatched` 增长超过发布前阈值（建议 0.5%）。

## 14. 验收标准（DoD）

### 14.1 功能验收

1. 管理端 CRUD 页面操作路径与行为不回归。
2. 前端可用单接口拿到 Item 聚合详情（支持 include）。
3. 聚合失败时自动回退旧接口，页面不白屏。

### 14.2 性能验收

1. 聚合详情接口 P95 < 300ms（预发、生产一致）。
2. 列表页首屏交互时间不劣化超过 10%。

### 14.3 质量验收

1. CI 强阻断项全部通过。
2. 发布灰度全档通过且无重大告警。
3. 回滚演练至少完成一次并记录复盘。

# TerraPedia 协作开发标准流程

## 1. 文档定位

这是一份仓库落地版 SOP，不是原则性说明文。

目标只有一个：

**把一次需求处理过程收敛成可执行、可检查、可回溯、可并行协作但不失控的标准流程。**

本文件默认适用于：

- 前端页面与交互修改
- 后端接口与服务逻辑修改
- 数据脚本执行与数据回填
- 本地联调与故障排查
- bug 修复
- review 与 git 提交

本文件不解决“业务方案是什么”，只解决“工作应该怎么推进才不跑偏”。

---

## 2. 仓库固定约束

### 2.1 默认环境

- 默认本地数据库：`terria_v1_local`
- 默认启动入口：`scripts/dev/start-local-stack.ps1`
- 默认停止入口：`scripts/dev/stop-local-stack.ps1`
- 默认本地配置模板：`scripts/dev/config/local-stack.config.example.json`

### 2.2 默认命令

启动本地三端：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\dev\start-local-stack.ps1
```

停止本地三端：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\dev\stop-local-stack.ps1
```

后端精确测试：

```powershell
cd back
mvn "-Dtest=ClassA,ClassB" test
```

后端常规测试：

```powershell
cd back
mvn test
```

管理端类型检查：

```powershell
cd data-query-app
pnpm run check
```

提交前最小检查：

```powershell
git status --short
git diff --cached --stat
```

### 2.3 默认事实优先级

遇到冲突时，默认按以下顺序判断事实：

1. 运行中的真实接口 / 页面行为
2. 当前数据库状态
3. 当前代码
4. 当前配置
5. 历史报告
6. 口头记忆或旧印象

## 3. 任务类型分流卡

进入任务后，先给任务归类，再决定默认推进顺序。

| 类型 | 默认顺序 | 典型风险 |
| --- | --- | --- |
| UI 任务 | 先数据链路，再模板，再样式 | 把数据缺失误判成样式问题 |
| 后端任务 | 先接口契约，再 service，再 mapper / SQL | 改了内部逻辑但外部返回不一致 |
| 数据任务 | 先现状统计，再历史报告，再 dry-run，再批量执行 | 共享字段混写、跑错数据库 |
| 联调任务 | 先启动链路，再接口，再页面，再数据库 | 实际跑的是旧进程或旧环境 |
| 混合任务 | 先锁影响范围，再拆分前后端与数据子任务 | 顺手改太多，提交失控 |

如果一个任务明显属于多种类型，按“风险最高的那个类型”走。

## 4. 强阻断总则

以下条目默认属于阻断条件。未满足时，**停止继续实施**，先补前置步骤。

### 4.1 阻断条件

- 未复述任务目标和成功标准
- 未核实环境事实
- 未锁定影响范围与目标文件
- 未定义验证方式
- 数据写入任务未核实现状
- 共享写入目标正在被其它任务写入
- 提交前未检查 staged 范围
- 未完成最小验证却准备结束或提交

### 4.2 最小验证定义

至少满足以下之一：

- 后端：编译通过 + 精确测试通过
- 前端：typecheck / 构建检查 / 页面运行态验证完成一项
- 数据任务：dry-run、数据库统计核对、样本核验完成一项

### 4.3 阻断后处理

发现阻断项后只能做下面三类动作：

1. 补环境核实
2. 缩小范围
3. 记录假设或例外原因

不得在阻断状态下继续扩散改动。

## 5. 标准执行流程

每一步都按同一模式执行：

- 进入条件
- 必须完成
- 可例外跳过
- 未满足时处理
- 产出物

### 第 1 步：任务进入

**进入条件**

- 已收到明确任务描述

**必须完成**

- 用一句话复述目标
- 定义成功标准
- 定义本次不做什么
- 判断任务类型

**可例外跳过**

- 无

**未满足时处理**

- 停止实施，只做澄清与环境核实

**产出物**

- `任务复述`
- `成功标准`
- `不在本次范围内的内容`

**检查清单**

- [ ] 我能用一句话说明用户要什么
- [ ] 我知道什么算完成
- [ ] 我知道本次不做什么
- [ ] 我知道这是前端 / 后端 / 数据 / 联调 / 混合中的哪类任务

### 第 2 步：环境核实

**进入条件**

- 已定义目标与成功标准

**必须完成**

- 找到页面 / 接口 / 脚本真实入口
- 确认当前数据来源链路
- 确认当前运行环境是否正确
- 确认可复用脚本、报告、测试

**可例外跳过**

- 纯文档任务可跳过运行态检查

**未满足时处理**

- 停止改代码，先查仓库和运行态

**产出物**

- `入口文件`
- `数据来源`
- `运行环境状态`
- `可复用资源`

**检查清单**

- [ ] 我知道真正的入口文件在哪里
- [ ] 我知道当前值来自前端、后端还是数据库
- [ ] 我知道当前本地到底在跑什么
- [ ] 我已经查过是否有现成脚本或报告可复用

### 第 3 步：澄清与建模

**进入条件**

- 环境事实已经核实

**必须完成**

- 明确影响范围
- 明确兼容要求
- 明确风险点
- 明确验证方式
- 把不确定项写成假设

**可例外跳过**

- 纯样式微调可简化兼容分析，但不能跳过验证方式

**未满足时处理**

- 不开始编码

**产出物**

- `影响范围`
- `兼容要求`
- `关键假设`
- `验证方式`

**检查清单**

- [ ] 我知道会影响哪些页面 / 接口 / 表 / 脚本
- [ ] 我知道哪些行为必须兼容
- [ ] 我知道这次最大的回归风险是什么
- [ ] 所有不确定项都已经写成明确假设

### 第 4 步：方案生成

**进入条件**

- 已完成任务建模

**必须完成**

- 先生成最小可行方案
- 明确改哪些层
- 明确不改哪些层
- 明确数据来源与优先级
- 明确验证顺序

**可例外跳过**

- 无

**未满足时处理**

- 不允许边写边想

**产出物**

- `方案摘要`
- `实施边界`
- `验证顺序`

**检查清单**

- [ ] 方案是最小可行而不是过度扩散
- [ ] 实施者拿到方案不需要再做关键决策
- [ ] 已经写清楚哪些层要改、哪些层不改
- [ ] 已经写清楚验证顺序

### 第 5 步：编码实施

**进入条件**

- 方案已收敛

**必须完成**

- 先锁定目标文件
- 只改相关文件
- 先修数据链路，再修 UI
- 若被 unrelated 问题阻塞，只做最小解除阻塞修复

**可例外跳过**

- 无

**未满足时处理**

- 停止扩散改动，先缩小边界

**产出物**

- 实际改动文件
- 被阻塞时的最小解除阻塞说明

**检查清单**

- [ ] 我知道当前允许修改哪些文件
- [ ] 我没有顺手改 unrelated 逻辑
- [ ] 若存在阻塞，我只做了最小修复
- [ ] 我能解释每个修改文件为什么必须改

### 第 6 步：测试与验证

**进入条件**

- 代码或数据已改完

**必须完成**

- 按固定顺序验证：
  1. 语法 / 类型 / 编译
  2. 精确测试
  3. 运行态接口验证
  4. 页面验证
  5. 数据计数或样本核验

**可例外跳过**

- 某项因既有仓库问题无法执行，但必须记录阻塞原因

**未满足时处理**

- 不得宣称完成

**产出物**

- 运行过的命令
- 通过项
- 阻塞项
- 与本次改动是否相关

**检查清单**

- [ ] 我至少做了最小验证
- [ ] 我知道哪些验证通过了
- [ ] 如果有失败，我知道失败是否与本次相关
- [ ] 我没有把“没跑”说成“已验证”

### 第 7 步：Review

**进入条件**

- 最小验证完成

**必须完成**

- 自检是否只改相关范围
- 自检是否引入兼容回归
- 自检是否还存在未处理边界条件
- 自检是否存在旧进程 / 旧脚本 / 旧缓存污染风险

**可例外跳过**

- 无

**未满足时处理**

- 不得进入提交

**产出物**

- `已解决问题`
- `残余风险`
- `未解决但与本次无关的问题`

**检查清单**

- [ ] 已解决项我能说清楚
- [ ] 残余风险我能说清楚
- [ ] 我知道哪些问题和本次无关
- [ ] 我知道用户是否需要刷新缓存、重启服务或重新跑脚本

### 第 8 步：Git 提交

**进入条件**

- 已完成 review

**必须完成**

- 检查 `git status --short`
- 检查 `git diff --cached --stat`
- 只提交本次任务相关文件
- commit message 用行为导向短句

**可例外跳过**

- 未要求 commit 时可跳过提交，但不能跳过 staged 范围检查

**未满足时处理**

- 不提交

**产出物**

- `commit id`
- `commit message`
- `提交文件范围`

**检查清单**

- [ ] 我检查过 staged 范围
- [ ] staged 里没有 unrelated 文件
- [ ] commit message 清楚说明行为
- [ ] 若未提交，我已明确说明原因

## 6. 多 agent / 并行 SOP

### 6.1 允许并行的场景

- 多块独立只读信息收集
- 多个彼此独立的文件修改
- 主线不依赖其结果即可继续推进
- UI 与脚本、文档与接口等写入边界天然分离

### 6.2 禁止并行的场景

- 同一文件
- 同一页面同一区块
- 同一数据库表同一字段
- 同一脚本长任务
- 同一端口对应的运行进程

### 6.3 并行前标准动作

并行前必须明确：

- 每个 agent 负责哪些文件
- 每个 agent 明确不能碰哪些文件
- 结果如何回收
- 是否存在共享写入目标

**并行前检查清单**

- [ ] 每个 agent 的写入边界已定义
- [ ] 不存在同一文件多人同时写
- [ ] 不存在同一字段多来源同时写
- [ ] 我知道主线什么时候依赖子任务结果

### 6.4 并行数据任务规则

如果是数据库写入任务，必须额外遵守：

- 先查有没有旧任务还在后台运行
- 先确认是不是同一张表、同一字段
- 有共享字段时必须串行
- 只有切分明确的分片任务才允许并行
- 出现锁等待或混写风险时立刻降级为串行

**数据并行前检查清单**

- [ ] 后台没有旧写入进程，或已明确停掉 / 等待完成
- [ ] 分片规则明确且互不重叠
- [ ] 写入来源单一
- [ ] 已定义失败后如何中止和回收

## 7. 特别场景处理卡

### 7.1 UI 问题

先查：

- 页面模板是否真的渲染了该字段
- 样式是否压住了布局
- 接口是否返回了数据
- 数据是字段缺失还是只是没显示

禁止：

- 一看到空白就先猜前端问题
- 一看到 404/空值就先猜后端问题

### 7.2 数据回填问题

先查：

- 当前数据库计数
- 历史回填报告
- 现有脚本能力
- 后台是否有长任务还在运行

推荐顺序：

1. 复用已有报告结果
2. 复用已有脚本
3. 先 dry-run
4. 小批量验证
5. 再扩大范围

禁止：

- 多来源同时写同一字段
- 不核实现状就直接开大批量

### 7.3 配置与启动问题

先看：

- `scripts/dev/config/local-stack.config.json`（仅本地使用，不提交）
- `scripts/dev/start-local-stack.ps1`

禁止：

- 在多个入口脚本里各自手改默认值
- 不确认旧进程是否还在就直接说“已切到新环境”

### 7.4 编译阻塞问题

如果被 unrelated 编译错误阻塞：

- 先判断是否与本次改动直接相关
- 若无关，只做最小解除阻塞修复
- 在最终结果中必须写明：
  - 阻塞文件
  - 错误信息
  - 为什么必须动它

禁止：

- 借编译阻塞顺手大面积重构 unrelated 模块

## 8. 紧急修复例外流程

紧急修复允许压缩流程，但不允许无流程。

### 8.1 可压缩项

- 方案展开深度
- 非关键验证范围
- 非必要的全量回归

### 8.2 不可压缩项

- 目标复述
- 影响范围确认
- 最小验证
- 风险说明
- staged 范围检查

### 8.3 紧急修复模板

```md
## 紧急修复例外记录
- 紧急原因：
- 用户影响：
- 本次跳过的步骤：
- 当前风险：
- 已完成的最小验证：
- 需要补做的验证：
- 预计补做时间：
```

如果不能写出这段记录，就不算真正的紧急修复流程。

## 10. 默认结论

本仓库默认执行原则只有一句话：

**先查清，再假设；先收敛，再修改；先验证，再提交。**

如果任何一步做不到，必须明确说明原因，而不是跳过不说。

## 二、技术栈对比：现有方案 vs 推荐方案

### 2.1 原始计划技术栈

原始项目计划推荐的是 **Astro（SSG 静态站点生成）** 路线：

| 层面 | 原计划选型 | 定位 |
|------|-----------|------|
| 前端 | Astro 5.x + Tailwind | 静态站点生成 |
| 搜索 | Pagefind | 构建时索引 + 客户端搜索 |
| 数据 | YAML/JSON 文件 + Content Collections | 数据即代码 |
| 后端 | 无（纯静态） | — |
| 数据库 | 无 | — |
| 部署 | Cloudflare Pages | 免费静态托管 |

### 2.2 实际实现技术栈

| 层面 | 实际选型 | 定位 |
|------|---------|------|
| 前端 | Vue 3 + Vite 6 + Tailwind + Pinia | SPA 单页应用 |
| 后台管理 | Nuxt 4 + Vue 3 + Tailwind + Pinia | SSR 管理后台 |
| 后端 | Spring Boot 3.2 + Java 17 + MyBatis Plus | REST API 服务 |
| 数据库 | MySQL 8 | 关系型数据库 |
| 部署 | 未配置 | — |

### 2.3 两条路线深度对比

| 维度 | 方案 A：保留现有栈（Vue+Spring Boot+MySQL） | 方案 B：重构为 Astro SSG（原计划） |
|------|-------------------------------------------|----------------------------------|
| **开发复杂度** | 🟡 中 — 三端分离，需维护 3 个项目 | 🟢 低 — 单一项目 |
| **已有代码利用率** | 🟢 100% — 无需重写 | 🔴 ~0% — 几乎全部重写 |
| **数据管理灵活性** | 🟢 高 — MySQL 支持复杂查询、动态 CRUD | 🟡 中 — YAML 文件，需重建才能更新 |
| **后台管理** | 🟢 已有 — Nuxt 后台管理几乎可用 | 🔴 无 — 需另建或用 CMS |
| **搜索能力** | 🟢 高 — SQL 多条件组合 + 模糊搜索 | 🟡 中 — Pagefind 纯客户端 |
| **运行环境要求** | 🟡 中 — 需 Java + MySQL + Node | 🟢 低 — 仅需 Node |
| **首屏性能** | 🟡 中 — SPA 需 JS 加载 | 🟢 高 — 纯静态 HTML |
| **SEO** | 🔴 低 — SPA 天然不友好 | 🟢 高 — SSG 天然友好 |
| **后续扩展（用户系统等）** | 🟢 容易 — 已有后端基建 | 🟡 中 — 需追加后端 |
| **Solo 维护成本** | 🟡 中 — 三端同步迭代 | 🟢 低 — 单项目 |
| **V1.0 到手时间** | 🟢 快 — 在现有基础上补全即可 | 🔴 慢 — 基本重写 |

### 2.4 推荐结论

**推荐方案 A：保留现有技术栈，在现有代码基础上完善至 V1.0。**

理由：
1. 三端代码已经完成 60%+ 的基础功能，重写成本过高
2. 你选择了 Spring Boot + MySQL 说明你对这套技术栈更熟悉
3. 有后台管理系统是长期优势，数据维护效率远高于编辑 YAML 文件
4. V1.0 目标是本地可用，SEO/部署问题可以后续解决（如接入 SSR 或 prerender）
5. 如果后续确实需要 SEO，可以单独做一个 Astro 层读取相同 API 生成静态页

### 2.5 推荐优化项（在保留现有栈的基础上）

| 优化项 | 说明 | 优先级 |
|--------|------|--------|
| 前端 SPA → Nuxt SSR/SSG | 前端也改用 Nuxt，与后台管理统一技术栈，获得 SEO 能力 | 🟡 可选（V1.5 考虑） |
| 加入 Flyway/Liquibase | 数据库版本管理 | 🟢 推荐 |
| API 文档（Swagger/SpringDoc） | 前后端协作标准化 | 🟢 推荐 |
| 简易登录（JWT） | 至少保护管理端 API | 🟢 推荐 |

## 六、本地开发环境要求

| 工具 | 版本要求 | 用途 |
|------|---------|------|
| Node.js | ≥ 18.x | 前端 + 后台管理 |
| npm/pnpm | 最新 | 包管理 |
| Java | 17 | 后端 |
| Maven | ≥ 3.8 | 后端构建 |
| MySQL | 8.x | 数据库 |
| VS Code / Cursor | — | IDE |

### 启动顺序

```bash
# 1. 启动 MySQL 并初始化数据库
mysql -u root -p < back/src/main/resources/schema.sql

# 2. 启动后端
cd back && mvn spring-boot:run
# 后端运行在 http://localhost:8888/api

# 3. 启动前端
cd front && pnpm install && pnpm run dev
# 前端运行在 http://localhost:5174

# 4. 启动后台管理
cd data-query-app && pnpm install && pnpm run dev
# 后台运行在 http://localhost:3001
```

## 八、后续展望（V1.0 之后）

V1.0 完成后，根据实际体验可以考虑的方向：

| 方向 | 说明 |
|------|------|
| **SEO 优化** | 前端接入 SSR（改用 Nuxt）或用 prerender 方案 |
| **线上部署** | Docker 容器化 + 云服务器部署 |
| **合成配方系统** | 可视化合成树，是 Wiki 类站点的核心差异化功能 |
| **Boss/NPC 模块** | 扩展数据模型，增加攻略内容 |
| **数据批量导入** | 爬虫/解析官方 Wiki 数据，快速填充 |
| **性能优化** | Redis 缓存、虚拟滚动、图片 CDN |
| **用户体系** | 用户注册、收藏夹、浏览历史 |

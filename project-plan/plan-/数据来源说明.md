# TerraPedia 数据来源说明

> 生成时间：2026-03-18
> 用途：说明当前项目的数据来自哪里、经过哪些处理、最终如何被前后端使用

---

## 1. 总体结论

当前项目的数据来源不是单一来源，而是一个分层链路：

```text
Terraria Wiki 官方数据
    ↓
本地抓取脚本
    ↓
本地清洗 / 映射 / 校验
    ↓
导入后端接口
    ↓
MySQL 数据库
    ↓
Spring Boot 后端 API
    ↓
前台 front / 管理端 data-query-app
```

也就是说：

- 原始物品数据主要来自 **Terraria Wiki 官方站**
- 当前页面运行时读取的主要是 **本地数据库中的导入结果**
- 分类、映射、图片等还叠加了本项目自己的本地规则和存储

---

## 2. 原始数据源

### 2.1 物品原始数据

当前仓库内的数据脚本显示，物品原始数据主要来自：

**Terraria Wiki 官方站**

对应脚本：

- `scripts/data/wiki-item-utils.mjs`
- `scripts/data/fetch-wiki-iteminfo.mjs`

其中 `wiki-item-utils.mjs` 中定义了默认 Wiki API：

```text
https://terraria.wiki.gg/api.php
```

抓取目标是 Wiki 的模块数据，例如：

```text
Module:Iteminfo/data
```

因此可以明确判断：

**当前项目的原始物品数据主要来自 terraria.wiki.gg 的官方模块数据。**

---

## 3. 中间处理数据源

抓下来的 Wiki 数据不会直接进入数据库，中间有一层本地数据加工。

### 3.1 数据清洗与标准化

相关脚本：

- `scripts/data/normalize-wiki-items.mjs`
- `scripts/data/validate-normalized-items.mjs`
- `scripts/data/validate-normalized-data.mjs`

作用：

- 将 Wiki 原始字段转换为当前项目的导入格式
- 筛掉不符合要求的数据
- 统一字段命名和结构
- 生成可导入的 JSON

### 3.2 分类映射修正

相关文件：

- `data/mappings/category-overrides.json`

作用：

- 对 Wiki 原始类别与本项目分类体系之间的映射进行本地修正
- 避免 Wiki 分类结构与本项目分类树不完全一致时导入错误

因此中间层可以理解为：

**Wiki 官方原始数据 + 本项目本地映射规则 + 本地清洗校验逻辑**

---

## 4. 导入数据源

标准化后的数据通过后端导入接口进入数据库。

相关脚本：

- `scripts/data/import-items.mjs`

默认导入地址：

```text
http://localhost:8888/api/items/import
```

说明：

- 导入脚本会把清洗后的 JSON 发送给后端
- 后端再将这些数据写入 MySQL
- 导入结果会写入导入报告目录

相关目录：

- `data/reports/import`

因此数据库里的物品数据，本质上是：

**官方 Wiki 数据经过本地清洗和映射后导入的结果**

---

## 5. 运行时主数据源

当前前台和管理端页面真正读取的数据源不是 Wiki，而是：

```text
MySQL 数据库
  -> Spring Boot 后端接口
  -> front / data-query-app
```

也就是说，页面运行时主数据源是：

**后端 API 返回的数据库数据**

### 5.1 前台数据来源

前台 `front` 通过 `/api/*` 请求后端接口读取：

- 物品列表
- 物品详情
- 分类树
- 统计信息
- 搜索联想

### 5.2 管理端数据来源

管理端 `data-query-app` 也通过 `/api/*` 读取后端接口：

- 物品 CRUD
- 分类管理
- 登录鉴权
- 图片上传
- 统计面板

因此：

**运行时主链路已经不依赖外部 Wiki，而依赖本地数据库。**

---

## 6. 分类数据来源

分类数据与物品数据不同，不是简单从 Wiki 原样搬运。

分类数据主要来自以下几部分：

- 本地数据库中的 `category` 表
- 初始化 SQL / Flyway migration
- 本地分类映射规则

因此当前分类树可以理解为：

**以本地定义的分类体系为主，Wiki 数据通过映射规则对齐到该体系。**

这意味着：

- 分类结构是项目自己维护的
- 不是完全由外部 Wiki 自动决定

---

## 7. 图片数据来源

图片来源目前是双来源结构。

### 7.1 原始图片地址

部分物品数据中可能已经带有图片地址字段，例如：

- `image`
- `imageUrl`

这些可能来自原始导入数据或历史数据。

### 7.2 管理端上传图片

管理端已经支持图片上传，上传后会进入 MinIO。

相关能力：

- 管理端上传图片
- 后端对象存储接口
- MinIO 存储

因此图片数据来源为：

**原始图片 URL + 管理端上传到 MinIO 的图片**

---

## 8. Mock / 演示数据来源

虽然当前主业务链路已经是真实数据库驱动，但仓库里仍存在少量 mock / fallback 数据。

### 8.1 前台 fallback 数据

例如：

- `front/src/views/HomePage.vue`

里面仍保留了：

- `mockItems`
- `mockCategories`

这些通常用于：

- 接口异常时兜底展示
- 本地开发时快速演示

### 8.2 管理端 SQL 查询页 mock 数据

管理端 SQL 查询页当前用的是本地 mock 路由，不是直接执行真实数据库 SQL。

因此需要明确：

**当前仓库中仍有少量 mock 数据存在，但它们不是主业务数据源。**

---

## 9. 当前数据来源分类总结

| 数据类型 | 当前主要来源 |
|----------|--------------|
| 物品原始数据 | Terraria Wiki 官方模块数据 |
| 清洗后导入数据 | 本地脚本生成的标准化 JSON |
| 运行时业务数据 | MySQL + Spring Boot API |
| 分类树 | 本地分类表 + 本地映射规则 |
| 图片 | 原始 URL + MinIO 上传 |
| 演示 / 兜底数据 | 少量本地 mock |

---

## 10. 最终结论

如果只用一句话概括当前项目的数据来源：

**原始数据主要来自 Terraria Wiki 官方数据，经过本地脚本清洗、映射和校验后导入 MySQL，前台和管理端运行时主要读取本地后端 API 返回的数据库数据。**

# TerraPedia 数据来源分析与真实 Demo 汇总（2026-03-26）

基线文档：`project-plan/数据来源方案_2026-03-25.md`

本次汇总基于 2026-03-26 的实际抓取与页面解析结果，不是伪造样例。结构化主数据主要来自 `terraria.wiki.gg`，版本校验数据来自 Steam 官方公告与 Terraria 官方论坛 RSS。

## 1. 结论先行

- `Terraria Wiki.gg` 仍然是最适合做生产主数据源的站点。
- 当前仓库已经验证可抓到 `items`、`npcs`、`projectiles`、`buffs`、`armor set bonuses`。
- 图片源也建议走 `wiki.gg`，但图片应独立于主数据处理，推荐使用 `File:` 文件页和 `imageinfo` API 解析真实图片 URL。
- 物品“获取方式”不能只做单个 `source` 字符串，应拆成多条 typed relation，例如 `crafting`、`drop`、`worldgen`、`mining`。
- `biome` / 群系数据是有的，但当前更接近“可解析页面源”，还不是项目里已经落地的结构化抓取脚本源。
- `Steam Community Announcements` 与 `Terraria Community Forums RSS` 都可直接获取，适合作为版本更新触发和官方复核源。
- `Fandom` 旧站仍可人工浏览，但脚本化抓取原始模块时会被 Cloudflare 拦截，不能作为稳定生产源。
- `ArmorSetBonuses` 目前拿到的是“组合关系 + 文本 key”，距离前台最终展示文案还差一层解析。

## 7. 图片源结论

### 7.1 结论

- 图片上游建议继续使用 `Terraria Wiki.gg`
- 图片获取路径不应依赖页面 HTML，而应走 `MediaWiki imageinfo API` 或 `Special:Redirect/file`
- Demo 阶段可以直接保存 `wiki.gg` 图片 URL
- 生产阶段更稳的做法是“先从 wiki.gg 解析图片，再同步到自有 MinIO / 对象存储”，最终前台使用自家 URL

### 7.2 本次已验证的图片解析方式

推荐接口：

```text
https://terraria.wiki.gg/api.php?action=query&titles=File:Iron%20Pickaxe.png&prop=imageinfo&iiprop=url&format=json&formatversion=2
```

稳定重定向入口：

```text
https://terraria.wiki.gg/wiki/Special:Redirect/file/Iron_Pickaxe.png
```

## 8. 获取方式 / 来源关系结论

### 8.1 结论

- 单个物品不应只保留一个字符串型 `source`
- 更合理的方式是把“获取方式”拆成多条 typed relation
- 当前最值得优先支持的来源类型至少包括：
  - `crafting`
  - `drop`
  - `worldgen`
  - `mining`
  - `shop`
  - `chest`
  - `event`
  - `quest`
  - `conversion`

### 8.2 真实 Demo 1: Copper Pickaxe

本次实测使用：

- `{{recipes|result=Copper Pickaxe}}`
- 词条页：`https://terraria.wiki.gg/wiki/Copper_Pickaxe`

解析到的真实结果：

- 主版本配方：
  - `Copper Bar x8`
  - `Any Wood x4`
  - 工作台：`Iron Anvil` 或 `Lead Anvil`
- 旧版本 / 3DS 变体：
  - `Copper Bar x12`
  - `Any Wood x4`

这说明：

- `Copper Pickaxe` 的“来源”应落为 `crafting`
- 配方本身应拆为独立结构，而不是写成一段说明文字

建议表示为：

```yaml
obtainedFrom:
  - type: "crafting"
    recipe:
      result: "Copper Pickaxe"
      amount: 1
      station: ["Iron Anvil", "Lead Anvil"]
      ingredients:
        - item: "Copper Bar"
          amount: 8
        - item: "Any Wood"
          amount: 4
      notes: "Desktop / Console / Mobile"
```

### 8.3 真实 Demo 2: Crimtane Ore

本次实测使用：

- 词条页：`https://terraria.wiki.gg/wiki/Crimtane_Ore`
- `parse` 返回的词条原文
- 页面中的掉落表行文本

词条原文可确认的真实信息：

- `Crimtane Ore` 主要来自 `Brain of Cthulhu`
- 在 `Crimson` 世界里，`Eye of Cthulhu` 也会掉落
- 该矿也会罕见生成在地下小矿脉
- 当高度低于 `0 ft` 时，需要 `Gold Pickaxe` 或更高采掘力；否则任意 pickaxe 即可开采

本次从掉落表实际清洗出的行文本：

- `Brain of Cthulhu 40–90 100%`
- `Creeper 5–12 / 5–7 / 2–4 66.67%`
- `Eye of Cthulhu 30–90 100%`

这说明：

- `Crimtane Ore` 不是单一来源
- 它至少同时属于：
  - `drop`
  - `worldgen`
  - `mining`

建议表示为：

```yaml
obtainedFrom:
  - type: "drop"
    sourceRefType: "boss"
    sourceRefName: "Brain of Cthulhu"
    quantity: "40-90"
    chance: "100%"
  - type: "drop"
    sourceRefType: "boss"
    sourceRefName: "Eye of Cthulhu"
    quantity: "30-90"
    chance: "100%"
    conditions: "Crimson worlds"
  - type: "drop"
    sourceRefType: "npc"
    sourceRefName: "Creeper"
    quantity: "5-12 / 5-7 / 2-4"
    chance: "66.67%"
    notes: "Different mode variants exist"
  - type: "worldgen"
    sourceRefType: "world"
    sourceRefName: "Underground veins"
    conditions: "Rare small veins underground"
  - type: "mining"
    sourceRefType: "world"
    sourceRefName: "Crimtane Ore vein"
    conditions: "Below 0 ft requires Gold Pickaxe or better"
```

### 8.4 对项目的数据建模建议

建议拆成两层：

1. `recipes` / `recipe_ingredients`
- 存“如何合成”
- 例如：结果物、配方数量、工作台、原料、版本差异

2. `item_acquisition_sources`
- 存“如何获得”
- 例如：掉落、采掘、世界生成、商店购买、宝箱、任务、事件

推荐字段：

- `item_id`
- `source_type`
- `source_ref_type`
- `source_ref_name`
- `quantity_min`
- `quantity_max`
- `chance`
- `station`
- `conditions`
- `notes`
- `source_page`
- `source_revision_timestamp`

### 8.5 对当前项目的直接判断

- 当前仓库还没有现成的“来源关系”抓取脚本
- 但从 `wiki.gg` 实测来看，至少“合成”和“掉落”已经能拿到真实结构或半结构数据
- 因此“来源关系”不是做不到，而是还没单独建采集管线

## 9. Biome / 群系结论

### 9.1 结论

- `biome` / 群系数据是有的，`wiki.gg` 上至少可以稳定访问并解析相关词条页
- 目前更适合将其归类为“页面解析源”，而不是已经验证好的模块化结构源
- 当前仓库只有 biome 的内容模型设计，没有现成 `fetch-wiki-biomes.mjs`
- 因此这部分可以做，但应按“解析词条页 + 抽字段”的方式推进

### 9.2 项目内现状

项目设计稿已经给出 biome 数据结构示例，见 `project-plan/03_架构设计.md`：

- `id`
- `name.zh / name.en`
- `layer`
- `type`
- `description`
- `enemies`
- `resources`
- `npcsPrefer`
- `uniqueDrops`
- `backgroundMusic`
- `icon`

这说明：

- biome 在产品设计上早就属于目标数据面
- 当前缺的是采集链路，不是展示概念

### 9.4 真实页面信息 Demo

#### The Crimson

本次解析到的真实信息包括：

- 是一个 `evil biome`
- 地表有 `Crimson grass`、`Crimson Tree`、`Vicious Mushroom`
- 有 `chasms`
- 每个 chasm 末端会有 `Crimson Heart`

- biome 类型：`evil`
- 相关子群系：`Underground Crimson`
- 特色资源 / 地貌：`Crimson Heart`、`Crimson Altar`、`chasms`

### 9.5 对 biome 数据源的判断

- 目前没有确认到像 `Iteminfo/Npcinfo/Projectileinfo` 这种现成模块化抓取入口
- 直接尝试 `CargoQuery` 的 `Biomes` 表没有得到可用结论，因此暂时不应把 biome 当成“现成结构化表源”
- 但页面内容本身较为规范，且几个核心群系页都有统一风格和明显章节，足够支撑一版页面解析方案

### 9.6 对项目的直接建议

- 第一阶段先覆盖核心 biome：
  - `Forest`
  - `Jungle`
  - `Desert`
  - `Snow biome`
  - `The Crimson`
  - `The Corruption`
  - `Hallow`
- 先抽稳定字段：
  - `name`
  - `alias`
  - `layer`
  - `type`
  - `description`
  - `linkedBiomes`
  - `resources`
  - `enemySamples`
  - `icon`
- 对“完整敌怪列表、完整资源列表、NPC 偏好”这类高波动信息，建议二阶段再做更细粒度同步

   Fandom

## 11. 对当前项目最直接的落地建议

- 物品、NPC、Projectile、Buff 已经足够支撑一版完整的“词条详情页 + 检索页 + 基础关系图”
- Buff 数据优先级应提高，因为它已经天然带了中英文本和 item / NPC 关联
- 图片接入建议单独做一条管线，不要混在纯文本主数据归一化里临时拼接
- 获取方式建议和 item 主表并行建模，不要退化成 `source = "Boss掉落/可挖掘/可合成"` 这种不可查询的混合文本
- biome 建议按“核心群系优先 + 页面解析抽稳定字段”推进，不要等完整结构化源出现后再做
- 版本更新检测建议每天跑一次 Steam + 论坛公告对比，每周或检测到新公告时再触发 wiki 全量重抓
- 套装加成可先入库“组合关系”，前台文案先不上，等 `textKey` 解析完成后再开放展示

# Batch39 分支与回滚策略（2026-03-24）

## 分支流设计
- `feat/front-article-compose-publish`
  - 目标：前端“编写并发布”功能实现（管理端页面与交互逻辑）。
- `chore/back-article-publish-verify`
  - 目标：后端能力核验与必要最小修复（仅在需要时改动）。
- `test/api-article-compose-publish`
  - 目标：接口联调验证与证据输出（请求/响应摘要、验收结论）。
- `feat/ui-article-compose`
  - 目标：UI 设计优化（仅样式/表现层改动）。
- `chore/repo-validation-batch39`
  - 目标：流程治理、分支策略与回滚规范文档（本分支）。

## 执行顺序（可落地）
1. 各流在各自分支独立提交，禁止跨流改动文件。
2. 每个分支至少 1 个可回滚提交（不要只留未提交改动）。
3. 先合并 `chore/back-article-publish-verify`（若无代码改动可跳过）。
4. 合并 `feat/front-article-compose-publish`。
5. 合并 `feat/ui-article-compose`。
6. 运行 `test/api-article-compose-publish` 的验收脚本，确认通过后再合并到目标主线。

## UI 流回滚方案（非破坏性）
前提：UI 改动在 `feat/ui-article-compose` 已提交。

1. 查看 UI 分支提交历史并定位需回滚提交：
```bash
git log --oneline --decorate --graph
```

2. 回滚单个 UI 提交（保留历史，新增反向提交）：
```bash
git revert <ui_commit_sha>
```

3. 回滚多个连续 UI 提交（按从新到旧顺序执行）：
```bash
git revert <newer_sha>
git revert <older_sha>
```

4. 若 UI 尚未合入主线，直接停止合并该分支即可（不需要改动主线历史）：
```bash
git switch <target_branch>
# 不执行 merge feat/ui-article-compose
```

## 流程约束
- 禁用破坏性操作：`git reset --hard`、`git checkout --`。
- 回滚优先使用 `git revert`，保证审计可追踪。
- 冲突处理只在对应流分支内完成，避免在主线临时修补。

# 前端信息架构与页面规划

## 1. 一级导航建议

建议前端一级导航直接按用户任务组织，而不是按数据库表名组织：

1. 物品
2. 配方
3. 群系
4. Boss
5. 事件
6. NPC
7. 阶段推进

## 2. 核心页面类型

### 2.1 列表页

每个一级域都应至少有：

- 列表
- 搜索
- 筛选
- 排序
- 分页或无限滚动

### 2.2 详情页

详情页不要只展示主表字段，而要展示完整关系。

#### 物品详情页

建议包含：

- 基本属性
- 主图与多图
- 获取方式
- 配方
- 相关群系
- 相关 Boss / NPC / 事件
- 来源与更新时间

#### Boss 详情页

建议包含：

- 基本信息
- 召唤条件
- 所属阶段
- 出现环境
- 掉落物
- 相关事件与进度关系

#### NPC 详情页

建议包含：

- 基本信息
- 是否 Town NPC
- 入住条件
- 商店内容
- 掉落
- 所在环境

## 3. 关键筛选维度

### 3.1 物品

- 分类
- 稀有度
- 阶段
- 来源类型
- biome

### 3.2 Boss / Event

- 是否 Hardmode
- 所属阶段
- biome
- 触发条件

### 3.3 NPC

- Town / Enemy / Boss
- biome
- 阶段

## 4. 首页结构建议

首页不要只做静态 Hero，建议直接体现 Terraria 的系统结构：

- 搜索入口
- 阶段推进入口
- 热门 Boss
- 热门物品 / 配方
- 群系导航
- NPC / 事件入口

这样首页会更像“系统地图”，而不是普通宣传页。

## 5. 用户路径建议

TerraPedia 最常见的用户路径通常有三条：

### 5.1 从物品出发

- 看到物品
- 想知道怎么获得
- 想知道怎么做
- 想知道在哪个阶段有用

### 5.2 从 Boss 出发

- 当前卡在某个 Boss
- 想知道召唤条件
- 想知道掉落
- 想知道打完解锁什么

### 5.3 从阶段出发

- 不知道当前该干什么
- 想知道这个阶段的推荐装备、Boss、材料与事件

所以阶段页很值得单独做，不要只放在文章里。

## 6. 推荐页面树

```text
/
/items
/items/:id
/recipes
/recipes/:id
/biomes
/biomes/:id
/bosses
/bosses/:id
/events
/events/:id
/npcs
/npcs/:id
/progression
/progression/:stage
```

## 7. 核心结论

前端信息架构的重点不是把表搬上页面，而是围绕用户问题组织：

- 这是什么
- 怎么获得
- 怎么做
- 在哪里出现
- 属于哪个阶段
- 会解锁什么

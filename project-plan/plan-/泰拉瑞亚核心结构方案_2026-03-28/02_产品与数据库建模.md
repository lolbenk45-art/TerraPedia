# 产品与数据库建模

## 1. 建模原则

Terraria 不适合用一个大表加几个长文本字段解决。

正确方向应该是：

- 主实体表负责存“本体”
- 关系表负责存“来源、配方、掉落、群系、事件、Boss、NPC 关联”
- 快照表负责存“来源可追溯”

核心结构：

- `items` 作为主实体
- `recipes / sources / images / biomes / bosses / events / npcs` 作为关系网络
- `progression` 作为阶段系统
- `snapshots` 作为溯源层

## 2. 主实体表建议

### 2.1 `items`

建议字段：

- `id`
- `internal_name`
- `name_en`
- `name_zh`
- `category_code`
- `rarity_id`
- `description`
- `tooltip`
- `damage`
- `defense`
- `use_time`
- `knockback`
- `buy_price`
- `sell_price`
- `stack_size`
- `image_url`
- `status`

### 2.2 `biomes`

建议字段：

- `id`
- `code`
- `name_en`
- `name_zh`
- `layer_type`
- `biome_type`
- `description`
- `icon_url`

### 2.3 `bosses`

建议字段：

- `id`
- `code`
- `name_en`
- `name_zh`
- `stage_code`
- `summon_item_id`
- `is_hardmode`
- `description`

### 2.4 `events`

建议字段：

- `id`
- `code`
- `name_en`
- `name_zh`
- `event_type`
- `is_hardmode`
- `trigger_conditions`
- `description`

### 2.5 `npcs`

建议字段：

- `id`
- `internal_name`
- `name_en`
- `name_zh`
- `npc_type`
- `is_town_npc`
- `is_boss`
- `damage`
- `defense`
- `life_max`
- `image_url`

## 3. 最关键的关系表

### 3.1 配方链

- `recipes`
- `recipe_ingredients`
- `recipe_stations`

### 3.2 物品获取链

- `item_sources`
- `item_images`
- `item_biomes`

### 3.3 Boss / Event / NPC 关联

- `boss_drops`
- `event_drops`
- `event_enemies`
- `npc_shops`
- `npc_drops`
- `npc_housing_requirements`

### 3.4 进度与来源

- `progression_stages`
- `progression_unlocks`
- `snapshots`

## 4. 必须避免的坏建模

下面这些做法后面都会出问题：

- 把掉落来源存进 `items.source_text`
- 把配方存成一大段字符串
- 把 biome 只当成标签
- 把 Hardmode 只写进说明文本
- 把图片只保留一个 URL 且没有来源信息

## 5. 推荐的产品能力

有了上面的结构后，前后端可以稳定支持：

- 查某个物品怎么获得
- 查某个材料能做什么
- 查某个 Boss 掉什么
- 查某个 biome 产什么资源
- 查某个阶段该打什么、做什么
- 根据 NPC 查询商店与入住条件

## 6. 核心结论

TerraPedia 的数据库设计不应围绕“页面”展开，而应围绕“实体 + 关系 + 阶段 + 快照”展开。

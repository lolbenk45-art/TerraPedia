# 泰拉瑞亚核心内容：产品与数据库建模版（2026-03-28）

> 目标：把 Terraria Wiki 中的核心系统，转换成适合 TerraPedia 这类产品的信息架构与数据库建模视角。
> 重点不是“介绍游戏”，而是回答一个更实际的问题：
> **如果要做一个可检索、可扩展、可维护的 Terraria 百科，核心数据应该怎么建。**

## 1. 建模原则

Terraria 不是单表能装下的游戏。

如果只用一个 `items` 表，把掉落、合成、群系、NPC、Boss、事件都塞进字符串字段里，后面会直接失控。更稳的做法是：

- 把“物品”当成核心实体
- 把“来源、配方、群系、Boss、NPC、事件、图片、快照”拆成关系表
- 把“世界推进”当成阶段系统，而不是备注文字

也就是说，TerraPedia 的数据库最好是：

- **主实体表 + 关系表 + 来源快照表**

而不是：

- **一个大 JSON 字段包打天下**

## 2. 核心数据域

基于 Terraria Wiki 的结构，最值得优先建模的数据域有 7 个：

1. 物品 `items`
2. 配方 `recipes`
3. 生物群落 `biomes`
4. Boss `bosses`
5. 事件 `events`
6. NPC `npcs`
7. 进度系统 `progression`

这 7 个域已经足够构成 TerraPedia 的第一版完整骨架。

## 3. 每个核心域该怎么建

### 3.1 物品 `items`

这是全站最核心的主表。

建议至少包含：

- `id`
- `internal_name`
- `name_en`
- `name_zh`
- `category_code`
- `rarity_id`
- `description`
- `tooltip`
- `damage`
- `defense`
- `use_time`
- `knockback`
- `buy_price`
- `sell_price`
- `stack_size`
- `status`
- `image_url`

这里的关键不是字段多，而是：

- **`items` 只存物品本体，不存复杂关系**

例如下面这些不应该直接糊在 `items.source` 里：

- “由僵尸掉落”
- “可在腐化之地获取”
- “用木头和石头在工作台制作”

这些都应该拆到关系表。

### 3.2 配方 `recipes`

配方是 Terraria 推进链最关键的关系域之一。

建议拆成两层：

- `recipes`
- `recipe_ingredients`

`recipes` 可以放：

- `id`
- `result_item_id`
- `result_quantity`
- `version_scope`
- `notes`
- `source_page`

`recipe_ingredients` 可以放：

- `id`
- `recipe_id`
- `ingredient_item_id`
- `ingredient_name_raw`
- `quantity_min`
- `quantity_max`
- `quantity_text`
- `sort_order`

如果制作站也要规范化，再加：

- `recipe_stations`

这比把整个配方存成一段文本强得多，因为后面你可以直接支持：

- 查某个物品怎么做
- 查某个材料能做什么
- 查某个工作台能做什么

### 3.3 生物群落 `biomes`

biome 不是附属分类，而是世界分层的关键实体。

建议主表字段：

- `id`
- `code`
- `name_en`
- `name_zh`
- `layer_type`
- `biome_type`
- `description`
- `icon_url`

关联表建议：

- `item_biomes`
- `biome_relations`
- `biome_resources`

原因很直接：

- 一个 biome 有自己的资源
- 一个物品可能和多个 biome 相关
- biome 之间还有“对应”“变体”“前后阶段关系”

如果不把 biome 单独拉出来，后续检索会非常弱。

### 3.4 Boss `bosses`

Boss 是推进门槛，不只是敌人目录。

建议主表字段：

- `id`
- `code`
- `name_en`
- `name_zh`
- `stage_code`
- `summon_item_id`
- `biome_id`
- `is_hardmode`
- `description`

关系表建议：

- `boss_drops`
- `boss_spawn_conditions`
- `boss_biomes`
- `boss_progression_links`

这样你才能支持：

- 某个 Boss 掉什么
- 某个 Boss 在什么条件下召唤
- 某个 Boss 属于哪个阶段
- 打完它会推进哪些系统

### 3.5 事件 `events`

`events` 和 Boss 不同，它更像“系统级战斗场景”。

建议主表字段：

- `id`
- `code`
- `name_en`
- `name_zh`
- `event_type`
- `is_hardmode`
- `trigger_conditions`
- `description`

关系表建议：

- `event_drops`
- `event_enemies`
- `event_conditions`

因为事件的价值不只是“什么时候发生”，而是：

- 产出什么
- 出现什么敌怪
- 和什么阶段绑定

### 3.6 NPC `npcs`

NPC 在 Terraria 里承担功能系统作用，不是单纯图鉴。

建议主表字段：

- `id`
- `internal_name`
- `name_en`
- `name_zh`
- `npc_type`
- `is_town_npc`
- `is_boss`
- `damage`
- `defense`
- `life_max`
- `banner`
- `image_url`

关系表建议：

- `npc_shops`
- `npc_housing_requirements`
- `npc_drops`
- `npc_biomes`
- `npc_buff_immunities`

对 TerraPedia 来说，Town NPC 和普通敌怪可以共用主表，但最好有类型字段，不然后续商店、住房、服务类数据会很难挂。

### 3.7 进度系统 `progression`

这是最容易被忽略，但产品层很重要的一块。

建议不要只靠文章写“前期/后期”，而应该有明确结构化数据：

- `progression_stages`
- `progression_requirements`
- `progression_unlocks`

例如阶段可以是：

- pre-hardmode
- early hardmode
- mechanical boss stage
- post-plantera
- lunar / endgame

这样你才能做：

- 按阶段浏览内容
- 推荐当前阶段该做什么
- 用阶段过滤物品、Boss、事件、NPC、群系信息

## 4. 最关键的关系表

如果只说最关键的关系表，我建议优先这 10 张：

1. `recipe_ingredients`
2. `recipe_stations`
3. `item_sources`
4. `item_images`
5. `item_biomes`
6. `boss_drops`
7. `npc_shops`
8. `npc_drops`
9. `event_drops`
10. `snapshots`

这 10 张表决定了 TerraPedia 是否只是“静态词条站”，还是“可查询的系统化百科”。

## 5. 图片该怎么建

图片不要只塞一个 `image_url` 字段就结束。

更稳的做法是：

- 主表保留一个当前主图字段，便于前端直接渲染
- 同时保留 `item_images` / `npc_images` / `boss_images` 这类关系表，支持多图、来源、主次图和尺寸信息

建议图片关系字段：

- `entity_type`
- `entity_id`
- `role`
- `provider`
- `source_file_title`
- `source_page`
- `source_revision_timestamp`
- `original_url`
- `cached_url`
- `width`
- `height`
- `content_type`
- `is_primary`
- `sort_order`

## 6. 快照与来源溯源

如果数据来自 wiki，就不要只存“结果”，还要存“来源快照”。

建议保留：

- `snapshots`

至少记录：

- `entity_type`
- `entity_id`
- `provider`
- `source_kind`
- `source_locator`
- `source_page`
- `source_revision_timestamp`
- `payload_json`
- `fetched_at`
- `parse_status`

这样做的价值是：

- 后续可追溯
- 数据异常时能回源排查
- wiki 更新后可以做差异比较

## 7. 产品层信息架构建议

如果按用户视角来组织页面，建议第一层不是“数据库表名”，而是：

1. 物品
2. Boss
3. NPC
4. 事件
5. 群系
6. 配方与制作
7. 阶段推进

然后每个详情页再挂关系模块。

例如物品详情页应天然包含：

- 基本属性
- 图片
- 获取方式
- 配方
- 相关 biome
- 相关 Boss / NPC / 事件
- 数据来源

这比只做一个“字段展示页”更符合 Terraria 本身的系统结构。

## 8. 数据优先级建议

### Phase 1：最小闭环

- `items`
- `recipes`
- `item_sources`
- `item_images`
- `biomes`

只要这五块做完整，用户已经能查：

- 这是什么物品
- 怎么获得
- 怎么合成
- 在哪里出现
- 长什么样

### Phase 2：推进系统

- `bosses`
- `events`
- `npcs`
- `progression_stages`

这一层做完，站点会从“物品库”升级成“游戏推进百科”。

### Phase 3：进阶规则

- `luck`
- 更细粒度的战斗规则
- 版本差异
- 多语言词条映射

## 9. 最终结论

从产品和数据库视角看，Terraria 最核心的不是某个单页，而是下面这条结构：

- `items` 作为主实体
- `recipes / sources / images / biomes / bosses / npcs / events` 作为关系网络
- `progression` 作为世界阶段与内容解锁框架
- `snapshots` 作为数据溯源层

如果 TerraPedia 要做得长期可维护、可检索、可扩展，最应该避免的就是：

- 把复杂关系写成备注文本
- 把多来源信息压成一个字符串字段
- 把阶段推进只放在文章里不做结构化

更合理的目标应该是：

- **主实体稳定**
- **关系表清晰**
- **图片可扩展**
- **来源可追溯**
- **阶段可查询**
